# VM 常见问题与排障

## 1. 是什么

这篇文档主要整理 Azure VM 在学习和测试过程中最常见的问题，以及对应的基础排查思路。

这里的 VM 包括：

- Windows VM
- Linux VM

你可以把这篇理解成：

> 一份面向初学者的 Azure VM 基础排障速查表。

它不是要一次覆盖所有复杂故障，而是先帮助你建立一个最重要的习惯：

> VM 出问题时，不要乱试，要按层检查。

---

## 2. 解决什么问题

这篇文档主要解决下面几类问题：

### 2.1 VM 创建成功，但不能正常使用
例如：

- 看起来部署成功了
- 但远程连不上
- 或者连上后状态不对

### 2.2 不知道先查哪里
很多初学者一遇到问题就会来回点 Portal，但没有排查顺序。  
这篇文档的重点就是帮你建立顺序。

### 2.3 分不清是计算问题、网络问题还是认证问题
Azure VM 的常见问题通常不是只有一个点，而是可能出在：

- 资源状态
- 网络路径
- NSG 规则
- 账号密码
- SSH Key
- 系统内部服务
- 成本和资源清理

---

## 3. 排障时先建立一个总思路

遇到 VM 问题时，建议优先按下面这个顺序排查：

### 第 1 层：资源是否真的存在且状态正常
先看：

- VM 是否创建成功
- VM 是否为 `Running`
- Public IP 是否存在
- 磁盘 / 网卡等资源是否正常挂载

### 第 2 层：网络是否可达
再看：

- IP 是否正确
- 本地网络是否能访问目标端口
- 网络路径是否通
- 是否本来就不该走公网

### 第 3 层：NSG / 防火墙 / 端口是否允许
重点看：

- Windows：3389
- Linux：22

还要看：

- 来源 IP 是否被限制
- 是否有拒绝规则
- 是否有系统内部防火墙限制

### 第 4 层：账号和认证方式是否正确
例如：

- Windows 用户名 / 密码
- Linux 用户名 / SSH Key / Password

### 第 5 层：系统内部服务是否正常
例如：

- RDP 服务
- SSH 服务
- Windows 启动状态
- Linux 系统负载
- 系统补丁或配置异常

---

## 4. 常见问题分类

可以先把 VM 常见问题分成几类来看：

### 4.1 创建类问题
- VM 创建失败
- 规格不可选
- 区域不支持
- 配额不足

### 4.2 连接类问题
- RDP 连不上
- SSH 连不上
- Connection timed out
- Authentication failed

### 4.3 访问类问题
- 有 IP 但无法访问
- 端口不通
- NSG 配置错误

### 4.4 系统类问题
- 系统卡顿
- 登录后黑屏
- SSH 登录后反应慢
- 应用服务起不来

### 4.5 生命周期与成本类问题
- 停机后仍计费
- 删除不彻底
- 遗留磁盘、公网 IP

---

## 5. 创建阶段常见问题

### 问题 1：VM 创建失败

常见原因包括：

- 区域容量不足
- 所选规格不可用
- 订阅权限不足
- 配额不够
- 用户名 / 密码不符合要求
- SSH Key 格式不正确
- 某些依赖资源创建失败

#### 排查建议
先看 Azure Portal 中的报错信息，重点关注：

- 是 Region 问题
- 是 Size 问题
- 是配额问题
- 还是认证参数问题

---

### 问题 2：某个 VM 规格选不了

常见原因包括：

- 当前 Region 不支持该规格
- 当前订阅该系列配额不足
- 当前区域容量不足

#### 排查建议
优先检查：

1. 是否换一个 Region
2. 是否换一个更小或更常见的 Size
3. 是否是订阅配额限制

---

### 问题 3：用户名或密码不通过

常见原因包括：

- 用户名是保留词
- 密码复杂度不满足要求
- 密码过于简单
- Windows / Linux 用户名规则不符合要求

#### 排查建议
重新设置：

- 更规范的用户名
- 满足复杂度要求的密码

---

## 6. 连接阶段常见问题

### 问题 4：Windows VM RDP 连不上

常见现象包括：

- 一直转圈
- 提示超时
- 提示无法连接
- 提示账号密码错误

#### 常见原因
- VM 没有 Public IP
- NSG 没有放通 3389
- 用户名 / 密码错误
- 本地网络限制 RDP
- VM 未完全启动
- 系统内部 RDP 服务异常

#### 排查顺序
1. 看 VM 是否 `Running`
2. 看是否有 Public IP
3. 看 NSG 是否允许 TCP 3389
4. 确认输入的用户名密码是否正确
5. 检查本地网络是否限制 RDP
6. 必要时检查系统级问题

---

### 问题 5：Linux VM SSH 连不上

常见现象包括：

- `Connection timed out`
- `Permission denied`
- `No route to host`
- 一直卡住无响应

#### 常见原因
- 没有 Public IP
- NSG 没有放通 22
- 用户名错误
- 私钥不匹配
- 使用了错误的 SSH 命令
- 本地网络限制 SSH

#### 排查顺序
1. 看 VM 是否 `Running`
2. 看是否有 Public IP
3. 看 NSG 是否允许 TCP 22
4. 看 SSH 用户名是否正确
5. 看私钥是否正确
6. 看本地终端命令是否写对

---

### 问题 6：Connection timed out

这个错误通常更偏向：

> 网络不通，而不是认证失败。

#### 常见原因
- IP 地址不对
- NSG 没放通端口
- 本地网络限制
- VM 没真正启动好
- 公网路径不可达

#### 排查重点
优先查：

- Public IP
- NSG
- 端口
- 本地网络

不要先怀疑密码。

---

### 问题 7：Authentication failed / Permission denied

这个错误通常更偏向：

> 认证失败，而不是网络不通。

#### 常见原因
- 用户名错误
- 密码错误
- SSH Key 错误
- 登录方式不匹配
- 私钥文件路径不对

#### 排查重点
优先查：

- 用户名
- 密码
- SSH Key
- 连接命令

---

## 7. 网络与安全相关常见问题

### 问题 8：有 Public IP，为什么还是连不上？

因为 Public IP 只说明“地址存在”，不说明“访问一定通”。

还要继续检查：

- NSG 是否放通端口
- 系统内部是否允许
- 用户名 / 密码 / Key 是否正确
- 本地网络是否有限制

---

### 问题 9：NSG 明明配了规则，为什么还是不通？

常见原因包括：

- 规则方向看错了（入站 / 出站）
- 端口写错了
- 协议写错了
- 优先级被其他规则覆盖
- 来源 IP 不匹配
- VM 网卡 / 子网实际绑定的 NSG 不是你看的那个

#### 排查建议
重点确认：

1. 规则绑在哪
2. 规则优先级
3. 来源地址范围
4. 目标端口
5. 协议是否正确

---

### 问题 10：为什么昨天能连，今天不能连？

常见原因包括：

- VM 被停止了
- Public IP 变化了
- NSG 被改了
- 本地网络变了
- 密码被改了
- SSH Key 使用错了
- 系统内部服务异常

#### 排查建议
不要假设环境“和昨天一样”，要重新核对：

- 当前 VM 状态
- 当前 IP
- 当前 NSG
- 当前认证方式

---

## 8. 系统内部常见问题

### 问题 11：登录后很卡或者很慢

常见原因包括：

- VM 规格过小
- 系统正在初始化
- 后台更新中
- 磁盘性能不足
- CPU / 内存占用高
- 网络本身延迟高

#### 排查建议
先看：

- VM Size
- CPU / Memory 指标
- 磁盘性能
- 是否刚创建完成

---

### 问题 12：Windows 登录后黑屏

常见原因包括：

- 系统初始化未完成
- 远程桌面会话异常
- 系统资源不足
- 图形界面加载异常

#### 排查建议
可以先：

- 等待一会儿
- 重启 VM
- 检查系统状态
- 观察是否反复出现

---

### 问题 13：SSH 登录成功但命令执行很慢

常见原因包括：

- VM 规格太小
- 系统负载高
- DNS 解析慢
- 磁盘性能不足
- 网络延迟高

#### 排查建议
可以先执行：

```bash
top
free -h
df -h
```

观察资源使用情况。

---

## 9. 生命周期与成本相关常见问题

### 问题 14：为什么我关机了还在收费？

因为要区分两种“关机”：

- 在操作系统里点关机
- 在 Azure Portal 里 `Stop / Deallocate`

很多时候只有真正释放计算资源后，计算费用才会停止。

#### 还可能继续计费的资源
即使 VM 停掉了，这些资源仍可能收费：

- 磁盘
- 部分公网 IP
- 备份
- 快照

---

### 问题 15：删除 VM 后为什么还有资源在？

因为删除 VM 不一定会自动删除所有配套资源。

常见遗留资源包括：

- OS Disk
- Data Disk
- Public IP
- NSG
- VNet（如果是独立创建的）
- 快照

#### 排查建议
实验结束后去资源组里看一遍资源列表，不要只删 VM 本体。

---

## 10. 一个基础排障清单

遇到 VM 问题时，可以先按下面这个清单快速过一遍。

### 10.1 资源状态
- VM 是否存在
- VM 是否 Running
- Public IP 是否存在
- NIC / Disk 是否正常

### 10.2 网络路径
- IP 是否正确
- 本地网络是否允许访问
- 是否能访问目标端口

### 10.3 NSG / 端口
- Windows 是否允许 3389
- Linux 是否允许 22
- 来源 IP 是否匹配
- 是否有更高优先级规则拦截

### 10.4 认证
- 用户名是否正确
- 密码是否正确
- SSH Key 是否正确
- 命令是否正确

### 10.5 系统层
- RDP / SSH 服务是否正常
- 系统是否完成启动
- 是否资源不足
- 是否系统异常

### 10.6 成本与清理
- 是否真正停止
- 是否有遗留磁盘
- 是否有遗留 Public IP
- 是否需要直接删整个资源组

## 11. 客户常问的问题

这一部分重点不是只记录客户怎么问，而是练习你该怎么回答。

---

### 问题 1：为什么 Azure VM 创建成功了还不能访问？

**简短回答：**  
因为“创建成功”只说明资源部署完成，不代表网络、端口和认证一定都已经正确。

**展开回答：**  
VM 真正可访问，通常还依赖：

- IP 是否正确
- 端口是否放通
- NSG 是否允许
- 账号和认证方式是否正确
- 系统内部远程服务是否正常

所以创建成功只是第一步，访问测试和排查同样重要。

---

### 问题 2：连不上 VM，一般先查什么？

**简短回答：**  
一般先查资源状态、IP、NSG 和认证方式。

**展开回答：**  
推荐顺序是：

1. 先看 VM 是否 Running
2. 再看有没有正确 IP
3. 再看 NSG 是否放通端口
4. 再看用户名 / 密码 / SSH Key
5. 最后再查系统内部服务

这样排查会比乱试更高效。

---

### 问题 3：为什么正式环境不建议直接开放 RDP / SSH 到公网？

**简短回答：**  
因为直接暴露管理端口到公网，安全风险更高。

**展开回答：**  
学习环境为了方便可以这样做。  
但正式环境通常会结合：

- VPN
- Bastion
- 跳板机
- 来源 IP 限制
- 更严格的身份验证和审计

来降低风险。

---

### 问题 4：为什么删了 VM 成本还是没完全下去？

**简短回答：**  
因为除了 VM 本身，磁盘、公网 IP、备份等资源也可能继续产生费用。

**展开回答：**  
客户经常会误以为删掉 VM 就等于所有成本都没了。  
实际上还要继续检查有没有遗留：

- Disk
- Public IP
- Snapshot
- Backup

所以清理测试环境时，通常建议查看整个资源组。

---

## 12. 你可以怎么对客户解释

如果客户问“为什么 VM 总是各种问题”，不建议一上来讲很多底层细节。  
可以先这样解释：

> Azure VM 本质上不是一个孤立资源，它通常和网络、IP、磁盘、安全规则、认证方式一起工作。  
> 所以一台 VM 能不能正常访问，不只是看它有没有创建成功，还要看网络路径、端口放通、账号认证和系统状态是不是都正常。  
> 一般排查时会按“资源状态 → 网络 → 端口 / NSG → 认证 → 系统内部”这个顺序来查，这样会更高效。

---

## 13. 实践建议

学习完这一篇后，建议至少做下面几个动作：

### 13.1 整理一次自己的真实故障记录
例如：

- RDP 连不上
- SSH Permission denied
- Public IP 看错
- NSG 没放通

把真实现象、原因和解决方法写下来。

### 13.2 故意做一次简单失败实验
例如：

- 把 3389 或 22 关掉
- 故意输错用户名
- 故意用错私钥

然后观察报错差异。

### 13.3 建立自己的排查顺序模板
例如：

1. 资源状态
2. IP
3. NSG
4. 认证
5. 系统内部

以后所有 VM 问题都先按这个顺序查。

### 13.4 检查一次资源组中的遗留资源
做完实验后，不要只看 VM 本体，要顺便看：

- Disk
- Public IP
- NSG
- VNet

## 15. 本篇总结

Azure VM 的常见问题，很多都不是“单点故障”，而是多个层面一起决定的：

- 资源状态
- 网络路径
- NSG / 端口
- 认证方式
- 系统内部服务
- 生命周期和成本管理

学习排障时，真正重要的不是背很多报错，而是建立一套稳定的排查顺序。

可以先记住这句话：

> **先看资源，再看网络，再看端口，再看认证，最后看系统内部。**
