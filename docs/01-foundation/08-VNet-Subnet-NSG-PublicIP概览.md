
# VNet / Subnet / NSG / Public IP 概览

## 1. 是什么

在 Azure 中，创建一台 VM 时，很多初学者最容易困惑的是：

- 为什么会自动出现 VNet？
- 为什么还会有 Subnet？
- 为什么 VM 会绑定 Public IP？
- NSG 又是控制什么的？

其实，这几个对象是 Azure 网络基础里最核心的一组概念。

你可以先把它们理解成：

- **VNet（Virtual Network）**：云上的虚拟网络
- **Subnet（子网）**：VNet 里面进一步划分的小网络
- **NSG（Network Security Group）**：控制流量访问规则的安全规则集
- **Public IP**：让资源可以被公网访问的地址

这篇文档的重点不是讲很深的网络原理，而是先建立一个最基础的整体认知：

> 一台 Azure VM 想要被访问，通常离不开网络、地址和访问控制这几个对象的配合。

---

## 2. 解决什么问题

这几个网络对象主要解决下面这些问题：

### 2.1 VNet 解决“资源放在哪个网络里”
Azure 里的资源不是直接裸奔在云上。  
很多资源都要先放进一个虚拟网络环境中。

### 2.2 Subnet 解决“同一个网络里怎么进一步分区”
一个 VNet 可以再划分成多个子网，用来区分不同类型资源。

### 2.3 NSG 解决“谁能访问谁”
不是所有网络流量都应该默认互通。  
NSG 用来控制允许或拒绝哪些访问。

### 2.4 Public IP 解决“资源怎么被公网找到”
如果你希望从本地电脑直接连一台 Azure VM，通常就需要 Public IP。

---

## 3. 先用一个类比理解

你可以先把这几个对象类比成下面这样：

- **VNet**：像一个园区
- **Subnet**：像园区里的不同楼层或不同区域
- **NSG**：像门禁规则
- **Public IP**：像对外可访问的门牌号

如果没有这个整体视角，很多初学者就会觉得这些对象很零散。  
但放到一起看，其实逻辑很清楚：

> 资源先放到网络里，再放到某个子网里，再决定要不要给它一个公网地址，最后再用 NSG 决定谁能访问它。

---

## 4. VNet 是什么

VNet（Virtual Network）是 Azure 中的虚拟网络。

可以简单理解为：

> 资源在云上运行时所在的私有网络环境。

很多 Azure 资源，尤其是 VM，通常都要放在某个 VNet 中。

### VNet 主要作用
- 给资源提供网络边界
- 让资源之间可以在虚拟网络中通信
- 为后续子网划分、NSG 绑定、私网访问打基础

### 初学阶段怎么理解
你先不用把 VNet 想得太复杂。  
学习阶段先把它理解成：

> Azure 里资源运行时所在的“网络大环境”。

---

## 5. Subnet 是什么

Subnet（子网）是 VNet 内部进一步划分出来的小网络。

可以简单理解为：

> 在一个大网络里再切出几个更小的分区。

例如一个 VNet 可以分成：

- 应用服务器子网
- 数据库子网
- 管理子网

### Subnet 的作用
- 对不同资源做网络分区
- 方便做隔离和管理
- 让不同类型资源按功能划分

### 初学阶段怎么理解
学习阶段先记住：

> VNet 是大范围，Subnet 是大范围里的小分区。

---

## 6. NSG 是什么

NSG（Network Security Group）是 Azure 中用于控制网络流量访问规则的安全规则集。

可以简单理解为：

> 一套决定“哪些流量能进、哪些流量能出”的规则。

### NSG 常控制什么
- 22 是否允许
- 3389 是否允许
- 80 / 443 是否允许
- 某个来源 IP 是否允许访问
- 某类流量是否要拒绝

### 初学阶段怎么理解
你可以先这样记：

> NSG 决定流量有没有资格通过。

---

## 7. Public IP 是什么

Public IP 就是 Azure 分配给资源、可以从公网访问的 IP 地址。

可以简单理解为：

> 云上资源对外公开的地址。

如果你希望：

- 用 RDP 连接 Windows VM
- 用 SSH 连接 Linux VM
- 用浏览器访问一台测试 Web 服务器

通常都需要有一个 Public IP，或者通过其他公网入口方式访问。

### Public IP 的作用
- 提供公网可达地址
- 让本地电脑或互联网流量可以找到目标资源

### 初学阶段怎么理解
先记住：

> 没有 Public IP，不代表资源不能访问；但如果你想从公网直接连它，通常需要它。

---

## 8. 这几个对象之间是什么关系

可以先用一句话记住：

- **VNet**：资源所在的大网络
- **Subnet**：VNet 里的小网络分区
- **NSG**：控制流量规则
- **Public IP**：提供公网访问地址

一个最简单的结构示意如下：

```text
VNet
└─ Subnet
   └─ VM
      ├─ Public IP
      └─ NSG（绑定在 NIC 或 Subnet）
```

也可以理解成：

1. VM 放在某个 VNet 中
2. 具体落在某个 Subnet 中
3. 如果需要公网访问，可以给它 Public IP
4. 最后由 NSG 决定流量是否允许通过

---

## 9. 创建 VM 时为什么会自动带出这些资源

很多初学者在创建 VM 后会发现资源组里突然多了很多对象，例如：

- VNet
- Subnet
- NIC
- Public IP
- NSG
- Disk

这是因为：

> 一台 VM 想真正工作起来，通常不仅仅需要“机器本身”，还需要配套的网络和安全资源。

所以 Azure Portal 在你没提前手动配置时，常常会帮你自动生成一套默认网络资源。

这对学习是好事，因为你可以借此观察：

- VM 是怎么和网络关联的
- Public IP 是怎么来的
- NSG 是怎么被挂上的
- VNet 和 Subnet 是怎么自动创建的

---

## 10. 学习阶段最常见的访问路径怎么理解

以最常见的“从本地电脑远程连接一台 Azure VM”为例，可以先这样理解：

### Windows VM
你的电脑 → Public IP → NSG 检查 3389 → VM

### Linux VM
你的终端 → Public IP → NSG 检查 22 → VM

也就是说：

- Public IP 决定你能不能找到这台资源
- NSG 决定这条访问请求让不让进
- VM 和系统决定进来后能不能真正登录成功

---

## 11. 一个最基础的例子

假设你创建了一台 Linux VM：

- 部署在某个 VNet 中
- 落在某个 Subnet 中
- 分配了 Public IP
- NSG 中允许 TCP 22

这时候你才有可能从本地终端执行：

```bash
ssh azureuser@<Public-IP>
```

如果其中任何一个条件不满足，例如：

- 没有 Public IP
- NSG 不允许 22
- VM 不在 Running 状态

那么你就可能连接失败。

这就是为什么这些对象一定要放在一起理解，而不能单独看。

## 12. 常见问题

### 问题 1：VNet 和 Subnet 有什么区别？

可以简单理解为：

- **VNet**：大网络
- **Subnet**：大网络里的小分区

VNet 更像网络范围，Subnet 更像网络内部的逻辑划分。

---

### 问题 2：NSG 和 Public IP 是不是一回事？

不是。

- **Public IP**：负责“地址”
- **NSG**：负责“规则”

你可以把它理解成：

- Public IP 告诉别人“去哪里找你”
- NSG 决定“找到了之后让不让进”

---

### 问题 3：有 Public IP 就一定能访问吗？

不一定。  
还要继续看：

- NSG 是否放通端口
- 系统服务是否正常
- 用户名 / 密码 / SSH Key 是否正确
- VM 是否处于 Running 状态

---

### 问题 4：没有 Public IP 就一定不能访问吗？

也不一定。  
没有 Public IP，只是不能直接从公网访问。  
但资源仍然可能通过下面这些方式访问：

- 内网访问
- VPN
- Bastion
- 跳板机
- 专线

所以“没有 Public IP”不等于“完全不能访问”。

---

### 问题 5：NSG 应该绑在 Subnet 还是 NIC？

两种都可以，具体要看你想在哪一层控制规则。

学习阶段先记住：

- 绑在 **Subnet**：更偏一组资源统一控制
- 绑在 **NIC**：更偏单台资源精细控制

---

## 14. 客户常问的问题

这一部分重点不是只记录客户怎么问，而是练习你该怎么回答。

---

### 问题 1：VNet 是干什么的？

**简短回答：**  
VNet 是 Azure 里的虚拟网络，用来承载云上资源的网络环境。

**展开回答：**  
很多 Azure 资源，尤其是虚拟机，通常都需要放在某个虚拟网络里。  
VNet 的作用就是给这些资源提供一个网络边界，让它们有地方通信、有地方做网络规划和隔离。

---

### 问题 2：为什么还要划分 Subnet？

**简短回答：**  
因为同一个大网络里，通常还需要进一步做资源分类和隔离。

**展开回答：**  
比如应用服务器、数据库、管理节点，很多时候不会全部混在同一个小网络里。  
Subnet 就是帮助我们在同一个 VNet 里进一步做分区和管理的。

---

### 问题 3：为什么有 Public IP 还要 NSG？

**简短回答：**  
因为 Public IP 解决的是“能不能找到”，NSG 解决的是“找到了之后让不让进”。

**展开回答：**  
客户最常见的误区就是觉得“有公网地址就一定能访问”。  
其实不是。  
能不能访问，还要看 NSG 是否允许对应端口和来源流量。

---

### 问题 4：是不是所有服务器都应该有 Public IP？

**简短回答：**  
不一定，很多正式环境并不会让所有资源都直接暴露到公网。

**展开回答：**  
学习环境为了方便测试，常常会直接给 VM 配 Public IP。  
但正式环境中，更常见的做法可能是：

- 通过 Bastion
- 通过 VPN
- 通过跳板机
- 通过私网访问

所以 Public IP 要不要给，要看访问方式和安全要求。

---

## 15. 你可以怎么对客户解释

如果客户第一次接触这些网络对象，不建议一开始讲太多术语。  
可以先这样说：

> 在 Azure 里，一台服务器不是单独存在的，它通常要先放在一个虚拟网络环境中。  
> VNet 可以理解成大的网络范围，Subnet 是里面的小分区，Public IP 是对外访问地址，NSG 则是控制哪些流量能进来的规则。  
> 所以如果想让一台云服务器真正可访问，不只是创建 VM 本身，还要把这些网络和安全对象一起看。

## 18. 本篇总结

VNet、Subnet、NSG、Public IP 是 Azure 中最基础、也最常一起出现的一组网络概念。

学习这一步时，真正重要的不是死记定义，而是建立一个整体认知：

- **VNet**：网络大环境
- **Subnet**：网络小分区
- **Public IP**：公网访问地址
- **NSG**：访问控制规则

可以先记住一句话：

> **资源先放进网络，再放进子网，需要公网访问时给地址，最后再用 NSG 控制谁能进。**
