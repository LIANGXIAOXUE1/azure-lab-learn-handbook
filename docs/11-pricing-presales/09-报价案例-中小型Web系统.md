# 报价案例：中小型 Web 系统

这篇文档的目标是：

> **用一个完整的中小型 Web 系统案例，把前面学过的 Azure 报价思路串起来。**

前面你已经看过：

- 报价整体流程
- 客户需求怎么收集
- 国际版和中国版怎么区分
- 常见机型怎么选
- CPU / GPU 场景怎么判断
- 客户需求怎么拆成 Azure 资源
- 常见漏项有哪些
- 报价备注怎么写

但如果这些知识只是分散地看，还不一定真正会用。

所以这一篇我们就用一个实际案例，完整走一遍：

1. 客户需求长什么样  
2. 先问什么问题  
3. 该拆成哪些 Azure 资源  
4. 为什么这么选型  
5. 哪些是主费用  
6. 哪些地方容易漏  
7. 最后的报价备注怎么写  

---

# 一、案例背景

假设现在有一个客户，需求是这样的：

## 客户类型
跨境电商公司

## 业务场景
客户要上线一套中小型 Web 系统，主要包括：

- 官网前台
- 管理后台
- API 服务
- 商品图片展示
- 用户、商品、订单等数据存储

## 使用区域
主要服务海外用户，计划部署在美国东部（East US）

## 目标
希望你给出一版 Azure 基础报价方案，用于预算和内部评估。

---

# 二、销售给到的初始需求

销售最开始给你的信息，通常不会很完整。  
例如这次只给了下面这些内容：

- 部署区域：美国东部
- 2 台 Web 服务器
- 1 台数据库服务器
- Web 用 4核16G
- DB 用 8核32G
- 每台都要磁盘
- 要做备份
- 要安全防护
- 希望按 1 年预留给一个参考价

如果你是新手，看到这里可能会想：

- 那我是不是直接去查 2 台 D4as v5 + 1 台 D8as v5 的价格就行？

其实还不行。  
因为这里虽然比“我要一个官网”详细，但仍然不是完整报价条件。

---

# 三、第一步：先判断这是哪一套 Azure

先做最前置的判断。

---

## 已知信息
- 区域：East US
- 服务海外用户
- 一年预留参考

## 判断
这属于：

> **Azure Global（国际版）**

---

## 为什么
因为：
- 部署在美国东部
- 不是中国大陆区域
- 一般按美元体系
- 常见优惠方式是 RI（Reserved Instance）

---

## 这一步对报价有什么影响
后面你要：

- 查 Azure Global 的价格
- 用国际版区域
- 价格单位按 USD
- 可以给按量 / RI 方案
- 如内部需要，再换算 RMB 展示

---

# 四、第二步：补齐需求信息

销售给的信息还不够完整。  
你需要继续补问。

---

## 至少还要确认这些问题

### 1）操作系统是什么
- Linux 还是 Windows？

### 2）数据库是什么类型
- MySQL？
- SQL Server？
- PostgreSQL？
- 还是数据库跑在 VM 上但软件客户自带？

### 3）磁盘多大
- 系统盘多大？
- 数据盘多大？
- 是 Standard SSD 还是 Premium SSD？

### 4）系统是否对公网开放
- 官网是否公网可访问？
- API 是否对外提供服务？

### 5）两台 Web 是否需要负载均衡
- 如果是生产系统，通常应该考虑

### 6）图片和附件放哪里
- 放 VM 磁盘？
- 还是用 Blob Storage？

### 7）是否需要 Redis
- 是否有缓存需求？

### 8）是否估算了流量
- 网站访问量大概多少？
- 图片访问量如何？
- 是否有下载内容？

---

## 假设补问后的结果如下

为了方便做案例，我们假设后续补齐的信息是：

- Azure Global 国际版
- 区域：East US
- Web 服务器操作系统：Linux
- 数据库服务器操作系统：Linux
- 数据库：MySQL，部署在 VM 上
- Web：2 台 D4as v5
- DB：1 台 D8as v5
- 每台系统盘：64GB Standard SSD
- Web 数据盘：128GB Standard SSD × 2
- DB 数据盘：512GB Premium SSD × 1
- 对外公网访问
- 2 台 Web 需要负载均衡
- 商品图片和附件希望单独存储
- 需要基础安全防护
- 需要基础备份
- 流量暂未精确统计，先按估算方式说明
- 需要一版按量和一版 1 年 RI 参考

这样信息就比一开始完整很多了。

# 五、第三步：把业务需求翻译成系统结构

现在不要急着查价。  
先把这个系统在你脑子里“画出来”。

---

## 这个系统大概长这样

### 入口层
- 用户从公网访问网站
- 流量先到统一入口
- 再分发到两台 Web

### 应用层
- 2 台 Web 服务器
- 跑官网前台、后台、API

### 数据层
- 1 台数据库服务器
- 跑 MySQL，存业务数据

### 文件层
- 商品图片、附件等单独存储

### 安全与运维层
- 基础安全防护
- 备份
- 可能还要考虑流量和监控说明

---

## 这时你就能把“客户要一套 Web 系统”翻译成：

> **公网访问 + 2 台应用服务器 + 1 台数据库 + 存储 + 负载均衡 + 安全 + 备份**

这就是报价的核心结构。

---

## 1）计算资源

### Web 层
- Linux VM D4as v5 × 2

### 数据库层
- Linux VM D8as v5 × 1

---

## 2）磁盘资源

### Web 系统盘
- 64GB Standard SSD × 2

### Web 数据盘
- 128GB Standard SSD × 2

### DB 系统盘
- 64GB Standard SSD × 1

### DB 数据盘
- 512GB Premium SSD × 1

---

## 3）网络资源

### 公网入口
- Public IP × 1

### 负载均衡
- Load Balancer × 1

---

## 4）存储资源

### 图片 / 附件存储
- Blob Storage × 1

---

## 5）安全与备份

### 基础安全

### 备份

## 6）用量类资源

### 出网流量
- 暂按估算或备注说明

### Blob 存储容量
- 暂按预估容量计或备注说明

# 八、为什么这样拆

这一段非常重要，因为它帮助你从“会列资源”变成“知道为什么这样列”。

---

## 1）为什么是 2 台 Web

因为客户给的是生产环境，而且系统要对外服务。

两台 Web 的好处一般有：

- 可以分担流量
- 单台故障时另一台仍可运行
- 更符合生产环境思路

---

## 2）为什么 Web 用 D4as v5

因为 Web 层通常用于：

- 官网前台
- 后台管理
- API 接口

这类中小型业务系统，4核16G 是很常见的起步规格。

它的特点是：
- CPU 和内存比较均衡
- 适合中小型 Web / API 业务
- 成本相对可控

---

## 3）为什么数据库用 D8as v5

因为数据库通常比应用层更吃资源，尤其是：

- CPU
- 内存
- 磁盘性能

而且这里数据库存的是：
- 用户
- 商品
- 订单
- 配置
- 后台数据

所以给数据库更高一档规格是合理的。

---

## 4）为什么数据库盘用 Premium SSD

因为数据库对磁盘性能更敏感。

如果数据库读写较频繁，Premium SSD 通常比普通 Standard SSD 更稳妥。

你可以先简单理解成：

> Web 可以先偏“够用”，数据库更要偏“稳”。

---

## 5）为什么要单独 Blob Storage

因为商品图片和附件如果全堆在 VM 数据盘里，会有几个问题：

- 扩容不灵活
- 不利于做统一文件管理
- 不利于后续接 CDN
- 应用层和文件层耦合太重

所以把图片和附件单独放对象存储，是更常见的云上做法。

---

## 6）为什么要 Load Balancer

因为 2 台 Web 如果都对外提供服务，就需要一个统一入口做流量分发。

否则用户不知道该访问哪台，系统也不是真正的双机对外。

---

## 7）为什么要 Backup

因为这是生产系统。  
有订单、有用户、有后台数据，不能只考虑“能跑”，还要考虑：

- 误删怎么办
- 服务器坏了怎么办
- 数据损坏怎么办

所以备份通常要纳入基础报价。

---

## 8）为什么要安全项

因为这套系统是公网开放的。

既然对外开放，就建议至少考虑基础安全能力，而不是完全裸奔。

---

# 九、这套报价里，哪些是主费用项

新手做报价时，还要学会看：

> **哪些是主要成本，哪些是辅助成本。**

---

## 通常主费用项是这些

### 1）虚拟机
- 2 台 Web VM
- 1 台 DB VM

这是最核心的大头。

### 2）数据库数据盘
特别是 Premium SSD 的数据盘，通常也是成本重点。

### 3）备份
如果保留周期较长，备份费用也会逐步明显。

### 4）流量
如果网站访问和图片下载较多，流量后期可能成为明显费用项。

---

## 辅助项通常包括
- Public IP
- Load Balancer
- Defender
- Blob Storage（前期容量不大时）
- 监控日志（如未大量开启）

这些单项不一定最大，但加起来也不可忽略。

---

# 十、这套方案里，最容易漏什么

这个案例特别适合练“漏项检查”。

---

## 1）磁盘
最容易只报 3 台 VM，却忘了：
- 3 个系统盘
- 3 个数据盘

---

## 2）Load Balancer
看到两台 Web，很容易只报 VM，不报流量分发入口。

---

## 3）Blob Storage
如果客户有商品图片，通常不能只盯着 VM 和 DB。

---

## 4）流量
很多人报了资源费，却没写出网流量说明。

---

## 5）Backup
客户可能没主动提，但生产系统强烈建议问。

---

## 6）安全项
公网 Web 系统通常至少要确认是否有 Defender / WAF / Firewall 需求。

---

# 十一、这时候要不要考虑 Redis？

这个问题很好，因为很多中小型 Web 系统介于“可要可不要”之间。

---

## 当前案例里怎么判断

如果客户只是：
- 中小型官网
- 后台管理
- 一般商品展示
- 访问量暂时不算大

那第一版基础方案里：

> **可以先不强制加入 Redis**

但要在“待确认项”里备注：

- 如客户后续访问量较高
- 登录状态、热点数据较多
- 希望减轻数据库压力

可再补充 Redis 方案。

---

## 这体现一个很重要的原则

报价不是“把所有能想到的资源都塞进去”，而是：

> **先给合理基础版，再按客户规模补增强项。**

---

# 十二、这套案例的报价输出结构可以怎么做

你不一定非要马上给客户看全部技术细节，  
但至少内部或销售版可以按下面方式整理。

---

## 方案 A：按量付费版

适合：
- 客户先看预算
- 暂未确定长期使用承诺
- 想先快速上线

### 包含
- Web VM × 2
- DB VM × 1
- 系统盘 / 数据盘
- Public IP
- Load Balancer
- Blob Storage
- Backup
- Defender
- 流量按估算或备注说明

---

## 方案 B：1 年 RI 版

适合：
- 客户生产环境较确定
- 资源会长期运行
- 希望压低长期成本

### 包含
- 同上资源清单
- 但 VM 等适合 RI 的部分按 1 年预留测算

---

## 可以给销售的展示方式
例如：

| 方案 | 月度预估 | 年度预估 | 备注 |
| ---- | ---- | ---- | ---- |
| 按量版 | xxx | xxx | 灵活，适合初期 |
| 1 年 RI 版 | xxx | xxx | 长期成本更优 |

# 十四、如果客户预算敏感，可以怎么做第二版方案

这是很实用的一步。

有些客户看完第一版，可能会说：

- 预算有点高
- 有没有更便宜一点的方案
- 能不能先上基础版

这时你可以做“标准版 + 低配版”。

---

## 标准版（更推荐）
- Web：D4as v5 × 2
- DB：D8as v5 × 1
- DB 数据盘：Premium SSD
- 含备份和基础安全

### 适合
- 正式生产
- 稳定上线
- 留一定性能余量

---

## 低配版（省钱版）
- Web：D4as v5 × 2
- DB：降为 D4as v5（仅在负载较轻时考虑）
- 数据盘容量适当下调
- 备份保留基础项
- 安全按最低必需项处理

### 适合
- 初期试运行
- 数据量较小
- 预算特别敏感
- 非关键业务或轻量生产

---

## 但要注意
如果客户真的是电商生产系统，数据库降配要非常谨慎。

你可以在备注里说明：

> 低配版适用于初期轻负载场景，若后续访问量、订单量或数据量增长，建议及时升级数据库规格及存储性能。

# 十五、这个案例教你的，不只是“报这一单”，而是一个方法

这篇案例真正想让你学会的，不是死记这个项目本身，而是下面这个方法：

---

## 第一步：先判断环境
是 Azure Global，还是 Azure 中国版。

---

## 第二步：收集并补齐需求
别只拿“2 台 Web + 1 台 DB”就开始查价。

---

## 第三步：先画系统结构
把业务系统拆成：
- 入口层
- 应用层
- 数据层
- 文件层
- 安全与运维层

---

## 第四步：再落成 Azure 资源
把每一层翻译成：
- VM
- Disk
- LB
- Storage
- Backup
- Defender
等具体资源。

---

## 第五步：检查主费用和漏项

- 哪些最贵

---

## 第六步：最后再做报价备注
把：
- 前提
- 已包含项
- 未包含项

写清楚。

# 十七、你可以怎么练这个案例

建议你不要只看一遍，可以这样练：

---

## 练习 1：自己口头复述
试着不看文档，自己讲一遍：

- 这是个什么系统
- 拆成了哪些层
- 用了哪些 Azure 资源
- 为什么这么选

---

## 练习 2：自己做“漏项检查”
把资源清单盖住，只看需求，问自己：

- 我会不会漏磁盘？
- 我会不会漏 LB？
- 我会不会漏存储？
- 我会不会漏备份和安全？

---

## 练习 3：自己再做一版“低配版”
比如试着调整：
- DB 从 D8 改 D4
- 不上 Blob Storage，改为先放本地盘（并说明风险）
- 暂不加 Defender
然后思考：
- 成本会怎么变
- 风险会怎么变
- 你会怎么备注

---

## 练习 4：反推客户系统结构
以后你看到任何历史报价单，都可以像这篇一样反推：

- 这像什么系统？
- 入口在哪？
- 应用层是哪些？
- 数据层是哪些？
- 哪些是配套项？

这是非常好的训练方法。

# 十八、这篇你要记住的核心结论

## 1. 一个中小型 Web 系统的 Azure 报价，通常不只是几台 VM
还会涉及磁盘、网络入口、存储、备份、安全和流量。

## 2. 先画系统结构，再列 Azure 资源，会比直接查价更清楚
这是新手最应该养成的习惯。

## 3. 数据库通常比 Web 更吃资源
所以规格和磁盘性能通常也更高。

## 4. 图片、附件、流量、负载均衡是 Web 系统里很常见的配套项
这些特别容易漏。

## 5. 报价最好至少准备按量版和 RI 版两种思路
这样客户更容易做预算比较。
