
# SAS Token 测试

## 1. 是什么

SAS Token（Shared Access Signature，共享访问签名）是 Azure Storage 中一种非常常见的临时授权方式。

你可以先把它理解成：

> 一个带有时间限制和权限限制的临时访问凭证。

在 Blob Storage 场景里，它最常见的作用就是：

- 让别人临时访问某个文件
- 让某个应用临时上传文件
- 让某个 URL 在一段时间内有效
- 不用直接暴露存储账号主密钥

所以学习 SAS Token 时，最核心的理解不是“它长什么样”，而是：

> 它是在控制“谁在什么时间内，以什么权限访问哪个资源”。

---

## 2. 解决什么问题

SAS Token 主要解决下面这些问题：

### 2.1 文件默认不能公开访问时，怎么临时共享
很多 Blob Storage 默认是私有的。  
这时如果你想让别人临时下载一个文件，就可以用 SAS Token。

### 2.2 不想把存储账号密钥直接给出去
直接暴露存储账号密钥风险很高。  
SAS Token 可以只授予有限权限，而不是把整把钥匙交出去。

### 2.3 想限制访问时间
例如：

- 这个链接只允许 1 小时内下载
- 这个上传权限只允许今天有效
- 过期后自动失效

### 2.4 想限制访问动作
SAS Token 不只是“给不给访问”，还可以控制：

- Read
- Write
- Delete
- List
- Create

也就是说，它可以细化到“能看、能传、能删到什么程度”。

---

## 3. 先建立一个最基础的理解

学习 SAS Token 时，建议先记住下面这句话：

> SAS 不是把文件变公开，而是给某个访问请求一个受限的临时许可。

也就是说：

- Blob 本身可能还是私有的
- 但只要你带着正确的 SAS Token 去访问
- Azure 就会在规定的时间和权限范围内允许你操作

这和“把容器公开”完全不是一回事。

---

## 4. SAS Token 常见使用场景

学习阶段先记住这些典型场景：

- 临时给别人一个下载链接
- 前端上传文件到 Blob
- 应用临时读写某个容器
- 给外部合作方一个限时访问地址
- 做测试链接，不想长期公开资源

对客户来说，SAS Token 常常对应的问题是：

- 文件怎么临时共享？
- 为什么链接可以今天打开，明天不行？
- 能不能只让别人下载，不能上传？
- 能不能设置有效期？

---

## 5. SAS Token 的核心组成

学习阶段不用一开始就记全部字段，但至少要理解下面几个关键维度：

### 5.1 资源范围
SAS 可以作用在不同层级，例如：

- 单个 Blob
- 一个 Container
- 整个 Storage Account（更大范围）

学习阶段最常见的是：

- 给单个 Blob 生成 SAS URL
- 给一个 Container 生成 SAS

---

### 5.2 权限范围
常见权限包括：

- Read
- Write
- Delete
- List
- Create

例如：

- 下载文件通常需要 **Read**
- 上传文件通常需要 **Write / Create**
- 查看容器中文件列表通常需要 **List**

---

### 5.3 时间范围
SAS 一般可以设置：

- 开始时间
- 过期时间

这意味着它不是永久有效的。  
过期后，即使 URL 还在，也通常无法继续访问。

---

### 5.4 URL 形式
你在 Portal 里生成 SAS 后，通常会得到一个带参数的 URL，类似：

```text
https://<storage-account>.blob.core.windows.net/<container>/<file>?<sas-token>
```

其中问号后面的那一长串参数，就是 SAS Token。

所以很多时候我们说 “SAS URL”，本质上就是：

> 原始 Blob URL + SAS Token 参数

---

## 6. 为什么 Blob URL 有时候打不开，而 SAS URL 能打开

这是学习 SAS 最关键的现象之一。

如果：

- Container 是 Private
- Blob 默认不允许匿名访问

那么你直接打开普通 Blob URL 时，通常会失败。

但如果你给这个 Blob 生成了一个带 Read 权限、在有效期内的 SAS URL，那么这个 URL 就可能访问成功。

所以你可以先记住：

> 普通 URL 表示资源地址，SAS URL 表示“带临时授权的资源地址”。

---

## 7. 怎么创建和测试

下面以 Azure Portal 为例，做最基础的 Blob SAS 测试。

---

### 7.1 前置准备

建议你先有下面这些资源：

- 一个 Storage Account
- 一个 Blob Container
- 一个已上传的测试文件，例如：
  - `hello.txt`
  - `test.png`
  - `sample.pdf`

---

### 7.2 找到目标 Blob

1. 登录 Azure Portal
2. 打开目标 Storage Account
3. 进入 `Containers`
4. 打开目标 Container
5. 选择一个已上传的 Blob

---

### 7.3 生成 SAS

在目标 Blob 页面，通常可以看到类似：

- `Generate SAS`
- 或共享访问签名相关入口

然后设置：

- 权限：勾选 `Read`
- 开始时间：可用默认或当前时间
- 过期时间：例如 1 小时后
- 协议：建议 HTTPS

然后点击生成。

---

### 7.4 获取 SAS URL

生成完成后，通常可以复制：

- Blob URL
- SAS Token
- 或完整的 SAS URL

建议直接复制完整的 SAS URL。

---

### 7.5 测试访问

把 SAS URL 粘贴到浏览器中测试：

#### 情况 A：可以打开
说明：

- Token 未过期
- 权限正确
- URL 正确
- 资源存在

#### 情况 B：打不开
常见原因包括：

- Token 已过期
- 权限不对
- URL 不完整
- 资源路径不对

---

## 8. 建议做的几个测试实验

### 8.1 测试 1：普通 Blob URL 与 SAS URL 对比

操作：

1. 复制普通 Blob URL
2. 浏览器打开
3. 再复制 SAS URL
4. 再打开

观察点：

- 普通 URL 是否失败
- SAS URL 是否成功

这个实验非常重要，因为它能帮助你直观看出：

> 资源地址和授权访问不是一回事。

---

### 8.2 测试 2：设置很短的过期时间

操作：

1. 生成一个有效期很短的 SAS，例如 5 分钟
2. 先访问一次，确认成功
3. 等到过期后再访问

观察点：

- 过期前可访问
- 过期后失效

这样你会更容易理解：

> SAS 最大的价值之一，就是临时性。

---

### 8.3 测试 3：只给 Read，不给 Write

操作：

- 生成只读权限的 SAS

观察点：

- 可以读取 / 下载
- 但不能做上传或修改类操作

这样你会更容易理解：

> SAS 是“按权限细粒度控制”的，而不是“要么全开，要么全关”。

---

### 8.4 测试 4：给 Container 生成 SAS

如果你后面想进一步理解，也可以测试：

- 针对整个 Container 生成 SAS

这样就能理解：

- Blob 级 SAS
- Container 级 SAS

的区别。

---

## 9. 这个实验想让你理解什么

SAS Token 测试，不只是生成一个链接，更重要的是理解下面这些事。

### 9.1 SAS 是授权机制，不是公开机制
它不是把文件永久公开，而是给特定访问加上一层临时授权。

### 9.2 访问控制可以细到“权限 + 时间”
你不只是决定“能不能访问”，还可以决定：

- 什么时候能访问
- 能访问多久
- 能做什么动作

### 9.3 SAS URL 才是实际用于共享的链接
很多时候客户说“给我一个文件下载链接”，真正交付出去的常常不是原始 Blob URL，而是 SAS URL。

### 9.4 过期是正常现象，不是系统坏了
如果链接昨天能用、今天不能用，很多时候不是文件坏了，而是 SAS 过期了。

---

## 10. 常见问题

### 问题 1：为什么普通 Blob URL 打不开？

常见原因包括：

- Container 是 Private
- Blob 不允许匿名访问
- 当前没有附带授权信息

#### 排查建议
先确认：

- 容器访问级别
- 是否需要 SAS
- 是否复制的是原始 URL 而不是 SAS URL

---

### 问题 2：为什么 SAS URL 一开始能打开，后来不能了？

最常见原因就是：

- **SAS 过期了**

也可能是：

- 资源被删了
- URL 复制不完整
- 权限设置不对

---

### 问题 3：为什么有 SAS 还是访问失败？

常见原因包括：

- SAS 过期时间已到
- 权限不足
- 资源路径不对
- URL 拼接错误
- 使用了错误的协议或参数

---

### 问题 4：SAS Token 能不能永久有效？

学习阶段不建议这么理解。  
SAS 的核心价值本来就是“临时、受限”。  
虽然可以把时间设得很长，但最佳实践通常不是把它当永久公开链接来用。

---

### 问题 5：SAS 和把容器公开有什么区别？

可以简单理解为：

- **容器公开**：很多人都能直接访问
- **SAS**：只有拿到正确链接的人，在规定时间和权限范围内能访问

所以 SAS 更细，也更适合临时共享。

---

## 11. 客户常问的问题

这一部分重点不是只记录客户怎么问，而是练习你该怎么回答。

---

### 问题 1：SAS Token 是干什么的？

**简短回答：**  
SAS Token 是 Azure Storage 中一种临时授权机制，用来控制谁能在什么时间内访问什么资源。

**展开回答：**  
如果客户要临时共享一个文件，又不想把整个存储账号暴露出去，那么 SAS 是非常常见的做法。  
它可以控制：

- 有效时间
- 可执行的操作
- 访问资源范围

所以它本质上是在做“受限授权”。

---

### 问题 2：为什么文件链接今天能打开，明天就不行了？

**简短回答：**  
很多时候是因为这个链接带的是 SAS，而 SAS 已经过期了。

**展开回答：**  
SAS 本来就是临时授权方式。  
如果设置的是 1 小时、1 天或 7 天有效，那么到期后链接失效是正常现象，不代表文件丢了。

---

### 问题 3：能不能只让别人下载，不让别人上传？

**简短回答：**  
可以。SAS 可以细化到权限级别，例如只给 Read，不给 Write。

**展开回答：**  
这也是 SAS 非常有价值的地方。  
你不是只能选择“开”或“不开”，而是可以细分到：

- 只读
- 只写
- 只列出
- 是否可删除

所以它比简单公开链接更灵活。

---

### 问题 4：SAS 安全吗？

**简短回答：**  
SAS 比直接暴露账号密钥安全得多，但是否安全仍取决于有效期、权限范围和分享方式。

**展开回答：**  
例如如果你生成了：

- 很长时间有效
- 权限很大
- 又随意传播

那风险还是会变高。  
所以 SAS 的安全关键在于：

- 最小权限
- 合理过期时间
- 受控分发

---

## 12. 你可以怎么对客户解释

如果客户第一次接触 SAS，不建议一上来讲参数细节。  
可以先这样说：

> SAS 可以理解成 Azure 存储里的临时访问凭证。  
> 它的作用不是把文件永久公开，而是在一定时间内、按指定权限，让某个人或某个系统访问这个文件。  
> 所以它特别适合做临时下载链接、临时上传授权和受限共享。

## 15. 本篇总结

SAS Token 是 Azure Storage 中非常重要的临时授权机制。

学习这一步时，真正重要的不只是“生成一个链接”，而是理解：

- SAS 是临时授权，不是永久公开
- 它可以控制时间、权限和资源范围
- SAS URL 是共享文件时非常高频的方式
- 链接失效很多时候是因为 SAS 到期，而不是文件出问题

可以先记住一句话：

> **SAS 解决的是“文件怎么被临时、安全、受控地访问”。**
