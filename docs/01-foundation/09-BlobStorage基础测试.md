
# Blob Storage 基础测试

## 1. 是什么

Blob Storage 是 Azure 中最常见的对象存储服务之一，属于 Azure Storage Account 下面的重要能力。

你可以先把它理解成：

> 一个用来存放文件、图片、视频、日志、备份包、文档等非结构化数据的云端存储空间。

这里的 “Blob” 可以简单理解为：

- 文件对象
- 二进制大对象
- 放在云上的文件数据

在学习阶段，你可以先把 Blob Storage 看成：

> Azure 里最常用的“文件上传与访问测试”入口之一。

---

## 2. 解决什么问题

Blob Storage 主要解决下面这些问题：

### 2.1 需要存放文件
例如：

- 图片
- PDF
- 压缩包
- 视频
- 日志文件
- 导出文件
- 备份文件

这些内容不适合直接放数据库时，常常会放到对象存储里。

### 2.2 需要一个可扩展的云端文件存储方式
相比本地磁盘，Blob Storage 更适合：

- 按需扩展
- 云上访问
- 结合应用使用
- 做下载链接或共享

### 2.3 需要让应用上传和读取文件
很多应用场景都会涉及：

- 用户上传头像
- 上传附件
- 上传图片
- 下载文档
- 静态资源托管

这些都常常会用到 Blob Storage。

### 2.4 需要做基础的数据共享与访问测试
学习 Azure 时，Blob Storage 是一个非常适合做入门实验的服务，因为它能帮助你理解：

- 存储账号
- 容器（Container）
- 文件上传
- URL 访问
- 公有 / 私有访问
- SAS Token

---

## 3. 先建立一个最基础的理解

学习 Blob Storage 时，建议先把下面几层关系分清楚：

- **Storage Account**：存储账号，是更上层的存储资源
- **Blob Service**：存储账号下的一种存储能力
- **Container**：类似文件容器 / 文件夹级逻辑分组
- **Blob**：具体的文件对象

可以先简单理解成：

```text
Storage Account
└─ Blob Container
   ├─ image1.png
   ├─ report.pdf
   └─ data.zip
```

所以后面你做实验时，通常不是“直接上传文件到 Azure”，而是：

1. 先创建 Storage Account
2. 再创建 Container
3. 再把文件上传成 Blob

## 4. Blob Storage 常见场景

学习阶段可以先记住这些高频场景：

- 网站图片存储
- 应用附件上传
- 日志归档
- 报表下载
- 静态资源托管
- AI / 数据处理输入文件存放
- 备份文件存放

对于客户来说，Blob Storage 经常对应的问题是：

- 文件放哪里
- 图片放哪里
- 附件怎么存
- 下载链接怎么做
- 文件能不能共享给别人

## 5. 创建前先理解什么

### 5.1 Blob Storage 依赖 Storage Account
你不能直接先有 Blob，再有存储账号。  
通常要先创建一个 Storage Account，然后在里面使用 Blob 服务。

### 5.2 Container 不等于本地文件夹
虽然 Container 看起来有点像文件夹，但它更像逻辑分组容器。  
学习阶段你可以先类比文件夹，但要知道它不是本地磁盘目录的完全等价物。

### 5.3 文件能不能访问，不只取决于“上传成功”
很多初学者会遇到：

- 文件上传成功了
- 但浏览器打不开
- 或别人无法访问

这通常和下面这些有关：

- 容器访问级别
- Blob URL 是否正确
- 是否使用了 SAS
- 是否本来就是私有访问

### 5.4 Blob Storage 会产生费用
费用通常和下面这些有关：

- 存储容量
- 访问次数
- 数据读取写入
- 出站流量
- 冗余类型

学习实验虽然不大，但也建议及时清理无用文件和测试资源。

## 6. 怎么创建和测试

下面以 Azure Portal 为例，做一个最基础的 Blob Storage 测试流程。

---

### 6.1 创建 Storage Account

1. 登录 Azure Portal
2. 搜索 `Storage accounts`
3. 点击 `Create`
4. 选择：
   - Subscription
   - Resource Group
5. 输入存储账号名称，例如：

```text
stazurelearnblob01
```

> 注意：Storage Account 名称通常要全局唯一，而且只能用小写字母和数字。

6. 选择 Region
7. 性能、冗余学习阶段可以先使用默认或基础配置
8. 点击 `Review + create`
9. 点击 `Create`

---

### 6.2 进入 Blob Service

创建完成后：

1. 打开这个 Storage Account
2. 找到 `Data storage`
3. 点击 `Containers`

---

### 6.3 创建一个 Container

1. 点击 `+ Container`
2. 输入容器名称，例如：

```text
test-container
```

3. 选择访问级别

常见会看到类似选项：

- Private（默认）
- Blob
- Container

学习阶段建议先用默认 **Private**，这样后面你会更容易理解“为什么直接 URL 打不开”。

4. 点击 `Create`

---

### 6.4 上传一个测试文件

进入刚创建的 Container 后：

1. 点击 `Upload`
2. 选择一个本地文件，例如：
   - `hello.txt`
   - `test.png`
   - `sample.pdf`
3. 点击 `Upload`

上传成功后，你就已经完成了最基础的 Blob 上传实验。

---

### 6.5 查看 Blob URL

上传成功后，点开这个 Blob 文件，通常可以看到它的 URL。

例如会类似：

```text
https://<storage-account-name>.blob.core.windows.net/test-container/hello.txt
```

这个 URL 非常重要，因为后面你会发现：

- 有时直接打开能访问
- 有时直接打开会报错
- 这就和访问控制有关

---

## 7. 怎么测试

学习 Blob Storage，建议至少做下面几组测试。

---

### 7.1 测试 1：确认 Storage Account 创建成功

检查：

- Storage Account 是否部署成功
- Region 是否正确
- 是否能进入 Containers 页面

---

### 7.2 测试 2：确认 Container 创建成功

检查：

- 容器名称是否正确
- 是否能正常打开容器详情页
- 是否能看到容器访问级别

---

### 7.3 测试 3：上传文件成功

检查：

- Blob 是否出现在容器中
- 文件名是否正确
- 大小是否正确

---

### 7.4 测试 4：尝试直接打开 Blob URL

把 Blob URL 复制到浏览器里测试。

可能会出现两种情况：

#### 情况 A：可以打开
说明当前访问策略允许直接访问。

#### 情况 B：打不开 / 报权限错误
这通常说明：

- 当前容器是私有的
- 需要授权方式访问
- 或后续需要 SAS Token

这个测试非常重要，因为它能帮助你理解：

> “文件上传成功” ≠ “文件默认对外公开可访问”

---

### 7.5 测试 5：删除 Blob

尝试删除刚上传的文件，观察：

- Blob 是否从容器中消失
- 删除操作是否立即生效

这样你会更清楚 Blob 的基础生命周期管理。

---

## 8. 这个实验想让你理解什么

Blob Storage 基础测试，不只是“上传一个文件”这么简单，更重要的是理解下面这些事。

### 8.1 Blob Storage 是对象存储，不是本地磁盘
它更适合文件对象、附件、图片、日志等非结构化数据的存储。

### 8.2 Storage Account 是上层资源
Blob 不是孤立存在的，它依赖存储账号和容器结构。

### 8.3 上传成功不代表默认可公开访问
访问能力和权限策略密切相关，这会直接引出后面的 SAS Token 学习。

### 8.4 Blob 的 URL 是后续共享和访问测试的基础
很多应用场景最后都会落到一个问题上：

- 这个文件 URL 能不能访问
- 谁能访问
- 访问多久有效

---

## 9. 常见问题

### 问题 1：为什么 Storage Account 名称总是创建失败？

常见原因包括：

- 名称已被别人使用
- 包含大写字母
- 包含不允许的特殊字符
- 长度不符合要求

#### 排查建议
尽量使用：

- 小写字母
- 数字
- 更有区分度的名字

例如：

```text
stlearnblob2025a
```

---

### 问题 2：为什么文件上传成功了，但 URL 打不开？

常见原因包括：

- 容器是 Private
- 当前 Blob 不允许匿名访问
- 没有使用 SAS Token
- URL 复制不完整

#### 排查建议
先确认：

- Container 访问级别
- 是否需要授权访问
- URL 是否完整正确

---

### 问题 3：Container 和 Blob 是什么关系？

可以简单理解为：

- **Container**：文件容器
- **Blob**：容器里的具体文件对象

也就是：

> Container 像一个逻辑容器，Blob 是里面的文件。

---

### 问题 4：Blob Storage 能不能当网盘用？

学习阶段可以先类比成“云上文件存放空间”，但严格来说它更偏对象存储服务，而不是传统个人网盘。

它更适合：

- 应用文件存储
- 系统附件存储
- 程序读写
- 下载链接
- 静态资源托管

---

### 问题 5：删除 Blob 后是不是就完全没了？

学习阶段可以先理解为：删除后该对象通常就不可访问了。  
但在更复杂场景里，还可能涉及：

- 软删除
- 版本控制
- 保留策略

这些可以后面再深入学习。

## 10. 客户常问的问题

这一部分重点不是只记录客户怎么问，而是练习你该怎么回答。

---

### 问题 1：Blob Storage 是干什么的？

**简短回答：**  
Blob Storage 是 Azure 里最常用的对象存储服务之一，适合存放图片、文件、附件、日志、备份等非结构化数据。

**展开回答：**  
如果客户的问题是“文件放哪里”“图片怎么存”“附件怎么下载”，很多场景下都可以考虑 Blob Storage。  
它适合做云上的文件对象存储，也容易和网站、API、数据处理流程结合起来。

---

### 问题 2：它和数据库有什么区别？

**简短回答：**  
数据库更适合存结构化数据，Blob Storage 更适合存文件对象。

**展开回答：**  
例如：

- 用户信息、订单数据 → 更适合数据库
- 图片、PDF、压缩包、日志文件 → 更适合 Blob Storage

所以 Blob Storage 不是拿来替代数据库的，而是解决“文件类数据怎么存”的问题。

---

### 问题 3：文件传上去后能直接给别人下载吗？

**简短回答：**  
可以，但要看访问策略，不一定默认就能公开访问。

**展开回答：**  
客户经常会误以为“上传后自动就有下载链接”。  
实际上还要看：

- 容器是不是公开的
- 文件是不是需要授权访问
- 是否要通过 SAS Token 控制时效和权限

所以“能不能下载”是访问控制问题，不只是上传问题。

---

### 问题 4：Blob Storage 安全吗？

**简短回答：**  
Azure 提供了比较完整的访问控制能力，但是否安全取决于实际配置方式。

**展开回答：**  
例如：

- 容器是否公开
- 是否使用 SAS Token
- 是否做了权限限制
- 是否使用私有访问方式
- 是否有日志和审计

所以安全不是“服务名字决定的”，而是“配置和访问控制决定的”。

---

## 11. 你可以怎么对客户解释

如果客户第一次接触 Blob Storage，不建议一上来讲太多技术术语。  
可以先这样说：

> Blob Storage 可以理解成 Azure 里专门用来放文件对象的云端存储。  
> 如果你要存图片、附件、PDF、压缩包、日志、备份文件，这类数据通常更适合放到 Blob Storage，而不是直接放数据库。  
> 它的重点不只是“能存”，还包括“怎么访问、谁能访问、是否需要授权访问”。

## 14. 本篇总结

Blob Storage 是 Azure 中最基础、也最常用的对象存储服务之一。

学习这一步时，真正重要的不只是“上传一个文件”，而是理解：

- Blob Storage 适合文件对象，不是结构化数据
- Storage Account 是上层资源，Container 是逻辑容器，Blob 是具体文件

可以先记住一句话：

> **Blob Storage 解决的是“文件放哪里”，而后续的访问控制决定“谁能打开它”。**
