# Azure 基础概念

## 1. 是什么

Azure 是微软提供的云计算平台（Cloud Platform），提供计算、存储、网络、数据库、安全、监控、AI、数据分析、物联网等多种云服务。

简单理解：

Azure 就是把原来需要企业自己购买、部署、维护的服务器、网络、存储、数据库等基础能力，放到云上，以服务的形式按需使用。

你不再需要先准备机房、采购硬件、安装环境，很多能力都可以直接在 Azure 上快速开通和使用。

---

## 2. 解决什么问题

Azure 主要解决的是企业 IT 和业务系统上线过程中常见的几个问题：

### 2.1 不想自己买服务器和机房
传统方式需要自己采购服务器、搭建机房、处理电力、网络、备份、扩容和维护。  
Azure 把这些底层基础设施托管起来，用户可以直接使用。

### 2.2 希望更快上线业务
如果要部署网站、API、数据库、内部系统，使用 Azure 可以比本地采购和部署更快。

### 2.3 希望按需扩容
业务流量小的时候少用资源，业务流量大时再扩容，不需要一开始就买很多硬件。

### 2.4 希望支持全球部署
Azure 在多个区域（Region）提供服务，可以把业务部署到更接近客户的位置。

### 2.5 希望提升运维、安全、监控能力
Azure 提供监控、日志、告警、身份认证、权限控制、备份、灾备等能力，方便统一管理。

### 2.6 希望直接使用 AI 和高级云服务
例如：

- Azure OpenAI
- Speech
- AI Search
- Document Intelligence
- Translation

这些都可以直接调用，而不需要从零开始搭建整套能力。

## 3. 核心理解：Azure 不只是“云服务器”

初学 Azure 时，最容易产生一个误解：

> 以为 Azure 只是“把服务器搬到云上”。

其实不是。  
虚拟机（Virtual Machine, VM）只是 Azure 中最基础的一类服务。Azure 更重要的价值在于，它提供的是一整套云平台能力。

常见能力包括：

- **计算**：Virtual Machine、App Service、AKS、Functions
- **存储**：Blob Storage、Disk、Files
- **网络**：VNet、Subnet、Load Balancer、Application Gateway、DNS
- **数据库**：Azure SQL、MySQL、PostgreSQL、Cosmos DB
- **身份与安全**：Microsoft Entra ID、RBAC、Key Vault
- **监控运维**：Azure Monitor、Log Analytics、Alerts
- **AI 服务**：Azure OpenAI、AI Services、Speech、AI Search
- **数据与集成**：Data Factory、Event Hub、Service Bus

所以更准确地说：

> Azure 是一个云平台，而不是单一的云服务器产品。

## 4. Azure 的基础资源模型

在真正开始创建资源之前，需要先理解几个最基础的概念。

### 4.1 Subscription（订阅）
Subscription 是 Azure 中计费和资源管理的基本边界。  
你创建的大部分资源，都属于某个订阅。

可以简单理解为：

- 它决定资源记在哪个账单下面
- 它也是权限和配额的重要边界之一

### 4.2 Resource Group（资源组）
Resource Group 是资源的逻辑分组容器。  
通常会把同一个项目、同一套实验、同一套应用相关的资源放在同一个资源组里。

例如：

- 一个测试环境一个资源组
- 一个项目一个资源组
- 一个练习实验一个资源组

### 4.3 Region（区域）
Region 是 Azure 数据中心所在的地理区域，例如：

- East US
- West Europe
- Southeast Asia

创建资源时通常要选择区域。  
区域会影响：

- 延迟
- 服务可用性
- 合规要求
- 成本
- 某些服务是否支持

### 4.4 Resource（资源）
在 Azure 中，虚拟机、存储账号、数据库、虚拟网络、公网 IP 等都属于资源。

也就是说，你在 Azure Portal 里创建的每一样东西，本质上都是一个 Azure Resource。

## 5. 怎么开始使用

最基本的使用步骤如下：

1. 注册 Azure 账号
2. 拥有一个可用的 Subscription（订阅）
3. 登录 Azure Portal
4. 创建一个 Resource Group（资源组）
5. 在资源组中创建所需资源，例如：
   - Virtual Machine
   - Storage Account
   - Virtual Network
   - SQL Database

常见操作入口包括：

### 5.1 Azure Portal
图形化界面操作，适合初学者快速理解资源关系和配置过程。

### 5.2 Azure CLI
命令行操作，适合批量管理、自动化脚本和日常运维。

### 5.3 PowerShell
适合习惯微软生态的管理员进行自动化操作。

### 5.4 ARM / Bicep / Terraform
用于基础设施即代码（IaC），适合标准化部署和团队协作。

学习 Azure 基础概念时，不需要一开始就创建很多复杂资源，先完成最简单的基础测试即可。

## 7. 学习 Azure 时最先要建立的认知

初学 Azure，不建议一开始就试图把所有服务都学完。  
更重要的是先建立下面这些基础认知：

### 7.1 Azure 是资源化管理的
你做的大部分事情，本质上都是在“创建、配置、连接、删除资源”。

### 7.2 Azure 是按服务组合来解决问题的
一个完整方案通常不是只靠一个服务，而是多个服务组合：

- App Service + SQL Database + Key Vault
- VM + VNet + NSG + Public IP
- Blob Storage + SAS + CDN
- Azure OpenAI + AI Search + Speech

### 7.3 Azure 学习不只是“会点 Portal”
真正有价值的是你能回答这些问题：

- 这个服务是什么？
- 它解决什么问题？
- 什么时候该用它？
- 跟别的服务有什么区别？
- 出问题时先查哪里？

### 问题 1：Azure 是不是只有虚拟机？

不是。  
Azure 不只是虚拟机，虚拟机只是其中一种最基础的计算资源。

Azure 更像是一个完整的云平台，除了虚拟机之外，还提供很多企业常用的能力，比如：

- 应用托管
- 容器
- 数据库
- 存储
- 网络
- 安全
- AI 服务

如果从客户价值来讲，重点不是“Azure 里有没有虚拟机”，而是：

> Azure 不只是把服务器搬到云上，而是把企业上云可能会用到的一整套能力都准备好了。

比如客户想快速上线一个网站，像公司官网、后台系统、简单 API，这时候不一定非要自己管理虚拟机，也可以直接用 **App Service** 来托管应用。  
这样可以少关心很多底层运维问题，部署速度也更快。

如果客户关心数据库，Azure 里也有很多托管数据库服务，比如：

- Azure SQL Database
- Azure Database for MySQL
- Azure Database for PostgreSQL

如果客户有文件、图片、附件、日志这类数据，也可以放到 **Blob Storage** 这种对象存储里。

如果客户后面还想接入 AI，Azure 还提供：

- Azure OpenAI
- Speech
- AI Search
- Document Intelligence

所以 Azure 的重点不是“只有虚拟机”，而是它能根据不同场景提供不同服务组合。

### 问题 2：Azure 是不是必须会命令行？

不是。  
Azure 完全可以先从图形界面开始学，不需要一上来就会命令行。

Azure 有自己的图形化管理界面，也就是 **Azure Portal**。  
对于初学者来说，Portal 很适合先理解这些基础概念：

- Subscription（订阅）
- Resource Group（资源组）
- Region（区域）
- 虚拟机
- 存储
- 网络

因为这些内容如果只看命令，很多时候不容易一下子建立整体认知。  
先在 Portal 里面实际创建、查看、删除资源，更容易理解 Azure 的资源关系。

命令行当然也很重要，比如：

- Azure CLI
- PowerShell
- Bicep
- Terraform

但这些更适合在你已经理解基本概念之后，再往自动化、批量管理、标准化部署方向继续学。

所以如果客户或者学习者问“是不是必须会命令行”，可以回答：

> 不需要，前期完全可以先用 Portal 学会核心概念，后面再逐步补 CLI 和自动化工具。

### 问题 3：Azure 和本地服务器的区别是什么？

最大的区别在于：  
本地服务器很多东西都要自己准备、自己维护，而 Azure 是云上按需使用。

如果是本地部署，通常要考虑很多前期投入，比如：

- 买服务器
- 买机房或托管机房
- 配网络设备
- 配电和环境
- 做系统安装和维护
- 做备份和扩容规划

这些事情都比较重，而且前期投入也比较大。

Azure 的好处是，很多基础设施能力平台已经准备好了。  
你可以根据需要快速创建资源，用多少开多少，不够再扩，不用时可以关停或删除。

从价值上说，Azure 比本地服务器更适合这些场景：

- 想快速上线系统
- 业务量有波动
- 不想一开始投入太多硬件成本
- 希望后续更方便接数据库、监控、安全、AI 等服务
- 希望支持异地或全球部署

简单来说可以这样解释：

- **本地服务器**：很多事情自己买、自己装、自己维护
- **Azure**：云上按需开通，扩展更快，组合服务更方便

所以 Azure 的优势一般体现在：

- 上线更快
- 扩容更灵活
- 全球部署更方便
- 运维和监控能力更完整
- 更容易和其他云服务组合

### 问题 4：用了 Azure 以后，是不是就完全不用管运维了？

不是。  
Azure 能帮你减少很多底层基础设施的运维工作，但不代表完全不需要运维。

这点要看你用的是哪一类服务。

如果你用的是 **虚拟机（VM）**，那你还是要关心很多事情，比如：

- 操作系统维护
- 账号和密码
- 远程连接
- 端口开放
- 补丁更新
- 应用部署
- 安全配置

也就是说，Azure 帮你解决的是“底层硬件和云基础设施”这部分，但虚拟机里面的很多系统和应用层管理，还是要自己负责。

但如果你用的是托管程度更高的服务，比如：

- App Service
- Azure SQL Database
- Functions

那你自己要管的内容通常会少很多，因为平台会帮你承担更多底层维护工作。

所以更准确的说法是：

> Azure 不是让运维消失，而是让很多运维工作更标准化、更自动化，并且可以根据服务类型把一部分工作交给平台。

对客户解释时，可以这样说：

> 如果你选择的是 VM，还是需要自己管不少内容；如果你选择的是平台型服务，运维压力会明显小很多。所以关键不是“上了 Azure 就不用运维”，而是“你选择什么服务模式，决定了你要自己管多少”。

### 问题 5：Azure 是不是很贵？

不一定。  
Azure 是按使用量和服务规格收费的，所以成本高不高，主要看你怎么设计和怎么用。

很多客户会直接问“Azure 贵不贵”，但这个问题不能一刀切回答。  
因为影响成本的因素很多，比如：

- 资源规格大小
- 使用时长
- 存储容量
- 网络流量
- 所在区域
- 是否做备份
- 是否开高可用
- 是否有监控、日志、安全等附加配置

举个简单例子：

- 如果只是一个小型测试环境，用比较小的规格，成本可能并不高
- 如果是生产环境，要求高可用、备份、监控、灾备、专线、安全加固，那成本自然会上来

所以不能简单说 Azure 一定贵，也不能简单说一定便宜。  
更合理的回答方式是：

> Azure 的成本取决于架构设计和使用方式。它的优势是按需使用、灵活调整，但如果资源选型过大、长期空转，或者架构设计不合理，成本也会被拉高。

如果客户更关注成本，你还可以继续往下展开：

- 先看是不是一定需要 VM
- 能不能用 App Service、Functions 这类更合适的服务
- 生产和测试环境是不是可以分开
- 晚上不用的资源能不能停机
- 存储和备份策略是不是合理

这样客户会感觉你不是只会说“云很好”，而是真的在帮他看成本。

### 11.2 创建一个测试资源组
