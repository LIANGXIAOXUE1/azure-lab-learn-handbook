# 客户需求信息收集清单

这篇文档解决的是 Azure 报价里最容易被忽略、但实际上最关键的一步：

> **报价前，先把客户需求信息收完整。**

很多人以为 Azure 报价最难的是算价格。  
其实实际工作里，更常见的问题是：

- 信息没问全
- 服务器用途不清楚
- 区域没确认
- 系统类型没确认
- 生产和测试环境混在一起
- 安全、备份、带宽这些配套项没问

结果就是：

- 报价反复改
- 容易漏项
- 容易报低或报高
- 客户后面质疑价格不准

所以这一篇的目标很明确：

1. 告诉你报价前到底要收哪些信息  
2. 告诉你哪些是必问项，哪些是补充项  
3. 给你一套可以直接发给销售或客户的收集模板  
4. 帮你减少后续反复沟通和返工  

---

# 一、先记住一句话：报价做不准，通常不是因为不会算，而是因为没问清楚

Azure 报价本质上是：

> **客户需求信息 → Azure 资源清单 → 价格汇总**

如果第 1 步信息就不完整，后面拆资源和报价一定会偏。

所以你做报价前，不要急着查价格。  
先确认：

- 客户到底要部署什么
- 这些服务器分别做什么
- 部署到哪里
- 是生产还是测试
- 有没有安全、备份、网络等附加需求

---

# 二、信息收集分 3 层看

你可以把要收集的信息分成 3 层：

## 第 1 层：基础必问项
这些问题不问清楚，基本没法准确报价。

## 第 2 层：资源补充项
这些问题决定资源是否完整、有没有漏项。

## 第 3 层：商务与备注项
这些问题影响报价呈现方式、计费方式和风险说明。

---

# 三、第一层：基础必问项（最重要）

这部分建议你优先掌握。  
如果销售只愿意先收最少信息，那至少把这部分拿齐。

---

## 1）客户要部署的是什么系统 / 业务

这是第一问，也是所有报价的起点。

你要知道客户到底要上的是什么，比如：

- 官网
- 商城 / 电商系统
- 管理后台
- ERP / OA / CRM
- APP 后端
- 数据库
- 文件存储系统
- AI 推理 / 图像识别
- 测试环境
- 迁移现有业务到 Azure

为什么这项最重要？

因为不同业务，资源结构差异很大。

例如：
- 官网：可能偏 Web + 存储 + CDN
- ERP：可能偏应用 + 数据库 + 内网访问
- 电商：可能要应用、数据库、缓存、负载均衡、备份
- AI 推理：可能涉及 GPU

---

## 2）部署区域（Region）

必须确认客户资源部署在哪个区域。

常见比如：

- East US
- Southeast Asia
- Japan East
- West Europe
- UAE North

为什么一定要问？

因为 Azure 不同区域价格可能不同，而且客户还可能有：

- 合规要求
- 用户访问延迟要求
- 数据驻留要求

如果区域没确定，报价只能算参考值。

---

## 3）国际版还是国内版

这个一定要分清。

因为 Azure 国际版和国内版不是同一套计费与运营体系。  
你要确认客户用的是：

- Azure Global（国际版）
- Azure 中国版（世纪互联）

如果这个没分清，后面价格体系和服务可选范围都可能不同。

---

## 4）生产环境还是测试环境

这个问题特别重要。

因为同一个系统，生产和测试的报价思路不一样。

### 生产环境
通常要求：
- 更稳定
- 更高可用
- 要留性能余量
- 更重视备份和安全
- 不适合过度压缩配置

### 测试环境
通常可以：
- 降一档规格
- 不一定配齐全部高可用资源
- 更偏成本控制

如果环境不区分，容易出现两个问题：

- 测试环境报太贵
- 生产环境报太低

---

## 5）服务器数量

要确认客户到底要几台计算资源。

例如：
- 1 台
- 2 台
- 3 台
- 多台集群

但这里只问数量还不够。  
一定要继续问下一项：**每台分别是做什么的。**

---

## 6）每台服务器用途

这是 Azure 报价里极其关键的一项。

不要只知道“客户要 3 台 VM”，一定要知道这 3 台分别跑什么。

例如：
- Web / 前端
- API / 应用服务
- 数据库
- Redis
- 日志服务
- 跳板机
- 代理 / 网关
- 测试机
- 文件服务

为什么用途这么重要？

因为用途决定选型。

比如：
- Web / 应用层：通常用通用型机型
- 数据库：通常需要更高 CPU / 内存 / 磁盘性能
- 代理 / Nginx：可能更偏 CPU
- AI 推理：可能涉及 GPU

---

## 7）每台服务器规格

至少要确认：

- CPU 几核
- 内存多大

例如：
- 4核 16G
- 8核 32G
- 16核 64G

如果客户已经指定机型，也要记录下来，比如：

- D4as v5
- D8as v5
- F4s v2

如果客户没给具体机型，你也要尽量拿到：
- 用途
- 并发规模
- 现有本地服务器配置

这样你才知道该怎么选型。

---

## 8）操作系统类型：Linux 还是 Windows

这个一定要问。

因为同规格的 VM：

- Linux 通常成本更低
- Windows 通常更贵

而且不同业务也常有偏好：

### 常见 Linux 场景
- Java
- Python
- Node.js
- PHP
- Docker
- Nginx
- MySQL

### 常见 Windows 场景
- IIS
- .NET Framework 老系统
- Windows 服务
- SQL Server
- 习惯 RDP 运维的客户

---

## 9）每台服务器是否需要数据盘，容量多大

新手很容易漏掉这一项。

很多客户说服务器规格时，只说 CPU 和内存，但不会主动说磁盘。

你要继续确认：

- 系统盘多大
- 要不要额外数据盘
- 数据盘多大
- 盘是普通 SSD 还是高性能 SSD

为什么？

因为系统盘通常只放操作系统，业务数据一般会放在数据盘里。

尤其是：
- 数据库
- 文件系统
- 日志系统
- 大量上传附件

如果数据盘没问，报价通常不完整。

---

## 10）客户希望按什么付费方式报价

这个是报价输出时必须确认的。

常见有：

- 按量付费（Pay as you go）
- 1 年预留（RI）
- 3 年预留（RI）
- 希望你同时给多个方案对比

很多客户最关心的其实不是“单价”，而是：

- 每月多少钱
- 一年多少钱
- 如果买预留能省多少

所以这个问题最好前面就确认。

---

# 四、第二层：资源补充项（防止漏项）

基础信息收完后，还要继续判断有没有配套资源需求。

这部分经常是报价漏项的来源。

---

## 1）是否需要数据库

你要确认客户数据库是：

- 已有，不在此次报价里
- 需要你一起报价
- 跑在 VM 上
- 用托管数据库服务

常见数据库类型：
- MySQL
- PostgreSQL
- SQL Server
- Azure SQL Database

需要继续问：
- 是生产还是测试库
- 预计数据量多大
- 有没有高可用需求
- 是否需要备份保留

---

## 2）是否需要缓存（Redis）

不是所有系统都要 Redis，但很多业务系统会用到。

典型场景：
- 登录状态
- 验证码
- 热点数据
- 会话缓存
- 降低数据库压力

你可以问：
- 有没有 Redis 需求
- 是客户明确提出，还是你根据架构建议补充

---

## 3）是否需要对象存储 / 文件存储

如果客户系统有下面这些内容，通常要考虑存储服务：

- 图片
- 视频
- 附件
- 导出文件
- 日志归档
- 共享文件

常见资源：
- Blob Storage
- Azure Files

---

## 4）是否需要公网访问

必须确认客户系统是不是对外开放。

例如：
- 官网
- 商城
- APP 接口
- 对外 API

如果需要公网访问，通常可能涉及：
- Public IP
- 域名
- HTTPS 证书
- 防火墙
- CDN
- DDoS

如果客户只内网使用，则网络设计和费用结构可能完全不同。

---

## 5）是否需要负载均衡

如果客户有多台应用服务器，就要进一步确认：

- 是否需要负载均衡
- 是否有高可用需求
- 是否需要对外统一入口

常见场景：
- 2 台或以上 Web / API 服务器
- 对外业务系统
- 不希望单点故障

---

## 6）是否需要 CDN

如果客户有大量静态资源，或者用户分布广，通常要考虑 CDN。

例如：
- 图片站
- 官网静态资源
- 跨境业务
- 下载类文件
- 视频内容分发

CDN 不一定每个项目都要，但很多公网项目适合补问。

---

## 7）是否需要备份

这个建议几乎都问。

你要确认：
- 是否需要 VM 备份
- 是否需要数据库备份
- 保留周期多久
- 是否需要跨区灾备

尤其生产环境，没有备份通常是不完整的。

---

## 8）是否需要安全项

你可以确认是否需要：

- Defender
- Firewall
- DDoS
- Key Vault
- 安全加固
- 合规要求相关服务

不是所有客户都会买齐，但你至少要问。  
特别是：
- 公网业务
- 电商
- 金融
- 医疗
- 政企客户

安全项往往比较敏感，不能默认忽略。

---

## 9）是否有流量估算

这一项非常重要，但经常没人主动提供。

你至少要知道一个大概范围：

- 网站月访问量
- APP 并发规模
- 每月出网流量估算
- 图片 / 视频下载量
- 接口请求量

为什么？

因为很多费用和流量相关，比如：
- 出网带宽
- 流量费
- CDN
- 网关服务
- DDoS 级别

如果完全没有流量概念，很多网络类报价只能给“资源费”，没法给完整总价。

---

# 五、第三层：商务与输出相关项

这些问题不一定影响技术架构，但会影响你最后怎么做报价。

---

## 1）客户希望看月度还是年度报价

常见情况有：

- 只看月度
- 要月度 + 年度
- 要年付预算
- 要做预算审批

建议默认准备：
- 月度总价
- 年度总价

---

## 2）是否需要多套方案对比

很多客户不会只看一套方案。

你可以提前问是否需要：

- 按量 vs 1年预留
- 标准版 vs 低配版
- 生产版 vs 测试版
- VM 方案 vs 托管服务方案

这样你一开始就知道要不要做双方案或三方案。

---

## 3）是否有预算范围

有些销售不愿意问预算，但如果能拿到，会非常有帮助。

例如客户说：
- 希望控制在每月 3000 美金以内
- 年预算约 10 万人民币
- 先做测试，成本越低越好

这会直接影响你：
- 配置建议
- 是否做高低配两版
- 是否优先推荐预留

---

## 4）是否有上线时间

这个问题看似不影响报价，其实很有用。

因为如果客户：
- 很快要上线
- 只是先看方案
- 正在做预算审批
- 还在比价阶段

你输出报价的详细程度和节奏都可以不同。

---

## 5）是否有特殊合规 / 数据驻留 / 安全要求

例如客户要求：
- 数据必须在某国家 / 地区
- 必须有日志审计
- 必须做密钥管理
- 必须启用防火墙
- 必须专线接入

这类要求会显著影响资源范围和价格。

---

# 六、最实用版本：新手报价前的 10 个必问问题

如果你现在还不熟，先死记这 10 个问题就够用了。

1. 客户要部署什么系统 / 业务？
2. 部署区域是哪里？
3. 国际版还是国内版？
4. 是生产环境还是测试环境？
5. 一共几台服务器？
6. 每台服务器分别做什么用途？
7. 每台需要什么规格（几核几 G）？
8. 操作系统是 Linux 还是 Windows？
9. 每台要不要数据盘？多大？
10. 要不要数据库 / Redis / 负载均衡 / CDN / 备份 / 安全？

如果这 10 个问题都拿到了，基本就能开始做第一版报价。

# 八、不同场景下的补充问题

有些场景，除了通用问题外，还要多问几句。

---

## 场景 1：官网 / 企业站
建议补问：
- 是否有后台管理系统
- 图片 / 视频多不多
- 是否需要全球访问加速
- 是否有下载内容
- 是否需要 CDN

---

## 场景 2：电商 / 商城系统
建议补问：
- 预计并发量
- 是否有订单、支付、库存系统
- 是否需要 Redis
- 是否需要多台应用服务器
- 是否要负载均衡
- 是否对数据库有性能要求

---

## 场景 3：内部管理系统 / OA / ERP
建议补问：
- 内部多少人使用
- 是否仅内网访问
- 是否需要 VPN
- 是否需要 AD / 权限体系
- 是否有数据库分离需求

---

## 场景 4：数据库上云
建议补问：
- 当前数据库类型
- 当前数据库版本
- 当前数据量
- 是否要求高可用
- 是否要求定期备份
- 是上 VM 还是上托管数据库

---

## 场景 5：AI / 图像 / 视频处理
建议补问：
- 是训练还是推理
- 是否明确需要 GPU
- 模型大小 / 数据量
- 使用时长是否连续
- 是否需要高性能存储

---

# 九、新手最容易漏问的 8 件事

下面这些特别容易忘，建议你重点检查。

## 1. 区域
没问区域，价格可能直接偏掉。

## 2. Linux / Windows
同规格价格可能差很多。

## 3. 数据盘
客户经常不会主动说，但实际又必须有。

## 4. 生产 / 测试环境
会直接影响规格和预算。

## 5. 数据库是否已包含
别默认客户“有数据库”就等于“本次不用报价”。

## 6. 备份
尤其生产环境，没问备份很危险。

## 7. 流量
很多网络和 CDN 费用都依赖流量估算。

## 8. 安全项
客户不一定主动提，但你不能完全不问。

# 十、如果客户信息不全，怎么先做初版报价

实际工作里，客户很多时候不会一次把信息给全。

这时可以这样做：

## 方法 1：先按已知信息出“基础版报价”
例如只先报：
- VM
- Disk
- 基础网络

同时备注：
- 备份、流量、安全、数据库等暂按未包含 / 待确认处理

---

## 方法 2：做“前提假设”
例如：
- 假设部署区域为 East US
- 假设系统为 Linux
- 假设每台包含标准 SSD 数据盘
- 假设出网流量未计入或按预估值计入

然后在备注里写清楚。

---

## 方法 3：给出“待确认项”
你可以单独列一栏：

### 待确认项
- 区域
- OS 类型
- 是否需要备份
- 流量预估
- 是否需要 CDN / 防火墙

这样销售会更容易知道下一步该补什么信息。

---

# 十一、一个好习惯：做报价前，先检查这张清单

你可以在自己心里快速过一遍：

- 业务类型清楚了吗？
- 区域清楚了吗？
- 国际版 / 国内版清楚了吗？
- 生产 / 测试清楚了吗？
- 服务器数量和用途清楚了吗？
- 规格和 OS 清楚了吗？
- 磁盘清楚了吗？
- 数据库 / Redis / 存储清楚了吗？
- 网络和访问方式清楚了吗？
- 备份和安全清楚了吗？
- 付款方式清楚了吗？

如果这些都清楚，报价就会顺很多。

---

# 十二、这篇你要记住的核心结论

## 1. Azure 报价前，先收需求，不要急着查价
信息不完整，算得再快也没用。

## 2. 最关键的不是“几台服务器”，而是“每台干什么”
用途决定选型，选型决定价格。

## 3. 新手至少要问清 10 个必问问题
先把基础信息拿齐，再谈准确报价。

## 4. 备份、安全、磁盘、流量是常见漏项
客户不一定主动提，但你要主动问。

## 5. 信息不全时，也可以先做带前提假设的初版报价
但一定要写清楚假设条件和待确认项。
