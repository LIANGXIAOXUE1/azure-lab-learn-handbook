# Azure 计算器使用说明

这篇文档主要解决一个非常实际的问题：

> **客户需求已经知道了，资源也拆出来了，那我到底怎么去查 Azure 价格？**

很多新手在这一步会出现几种典型情况：

- 知道客户要 2 台 Web + 1 台数据库，但不会去哪里查价
- 打开 Azure 官网后，页面很多，不知道从哪里开始
- 看到计算器选项太多，不知道该填什么
- 查出来价格后，不知道怎么整理进报价单
- 国际版和中国版查价入口又容易混

所以这一篇的目标是：

1. 帮你分清国际版和中国版分别去哪里查价  
2. 帮你建立“先拆资源，再逐项查价”的方法  
3. 帮你掌握 VM、磁盘、存储、Redis、备份等常见资源的基础查价思路  
4. 告诉你查价时最容易出错的地方  
5. 教你怎么把查到的结果整理进报价单  

---

# 一、先记住一句话：Azure 查价不是“搜一个总价”，而是“按资源逐项查”

Azure 没有一个按钮叫：

- 官网多少钱
- 电商系统多少钱
- ERP 系统多少钱

Azure 查价的正确方法是：

> **把客户系统拆成一项项 Azure 资源，再分别查每一项的价格。**

例如一个中小型 Web 系统，可能拆成：

- Linux VM D4as v5 × 2
- Linux VM D8as v5 × 1
- 系统盘 × 3
- 数据盘 × 3
- Blob Storage × 1
- Load Balancer × 1
- Backup × 1
- Defender × 3
- 流量估算 × 1

然后再把这些价格汇总。

---

# 二、先分清：国际版和中国版查价入口不一样

这是第一步，必须先判断清楚。

---

## 1）Azure Global（国际版）查价入口

常见入口包括：

### Azure 官方价格计算器
https://azure.microsoft.com/zh-cn/pricing/calculator/

你可以用它来查：
- VM
- Disk
- Storage
- Database
- Redis
- CDN
- Backup
等常见服务价格。

---

## 2）Azure 中国版（世纪互联）查价入口

常见入口是 Azure 中国官网对应服务价格页，例如：
https://www.azure.cn/pricing/

中国版很多时候不是和国际版完全一样的交互式计算器体验，  
有时更常见的是：

- 服务价格详情页
- 各产品定价页
- 对应中国区价格说明页

所以你要记住：

> **国际版常用全球计算器，中国版更多看中国官网价格页。**

---

# 三、新手查价前的正确顺序

你不要一上来就开网页乱点。

建议你永远按这个顺序做：

---

## 第一步：先确认是哪一套 Azure
- Azure Global
- Azure 中国版（世纪互联）

---

## 第二步：先把资源清单列出来
先写明：
- 什么资源
- 数量
- 区域
- 规格
- 系统
- 付费方式

---

## 第三步：再逐项查价格
按：
- 计算
- 磁盘
- 数据库
- 存储
- 网络
- 安全
- 备份

依次查。

---

## 第四步：把结果统一整理到报价表
不要查一个记一个、到处散落。

最好有一个固定表格，把：
- 资源名
- 规格
- 数量
- 单价
- 月价 / 年价
- 备注

统一写进去。

---

# 四、查价前你至少要准备好的信息

如果这些信息没有准备好，你打开计算器也会乱。

---

## 1）区域（Region）
例如：
- East US
- Southeast Asia
- Japan East
- 中国北部 3

---

## 2）资源规格
例如：
- D4as v5
- D8as v5
- 64GB Standard SSD
- 512GB Premium SSD

---

## 3）操作系统
例如：
- Linux
- Windows

---

## 4）数量
例如：
- Web VM × 2
- DB VM × 1

---

## 5）付费方式
例如：
- Pay as you go
- 1 年 RI
- 3 年 RI
- CPP

---

## 6）是否有用量估算
例如：
- 出网流量多少
- Blob 容量多少
- 备份容量多少

---

# 五、国际版 Azure 计算器怎么用：整体思路

下面先讲国际版，因为国际版的计算器更常用于直接配置。

---

## 国际版常见使用流程

### 第 1 步：打开 Azure Pricing Calculator
进入：
https://azure.microsoft.com/zh-cn/pricing/calculator/

---

### 第 2 步：搜索你要的资源
例如搜索：
- Virtual Machines
- Managed Disks
- Blob Storage
- Azure Database for MySQL
- Load Balancer
- Redis
- Backup

---

### 第 3 步：把对应资源加到计算器
你不是查一次“整个项目”，而是把每个资源逐项加入。

---

### 第 4 步：逐项配置参数
例如 VM 要选：
- Region
- OS
- VM Series / Size
- Number of instances
- Pricing option

磁盘要选：
- Disk type
- Capacity
- Quantity

---

### 第 5 步：记录结果
把每项月度价格、年度价格、按量 / RI 价格记到你的报价表。

---

# 六、查 VM 价格的基本方法

这是最常见的一项。

---

## 1）你要准备什么信息

查 VM 前，至少先知道：

- 区域
- Linux / Windows
- VM 规格
- 数量
- 付费方式

例如：
- Region: East US
- OS: Linux
- VM Size: D4as v5
- Quantity: 2
- Pricing: Pay as you go / 1-year RI

---

## 2）在计算器里通常怎么选

以 Virtual Machines 为例，一般会看到类似选项：

- Region
- Operating System
- Type / Instance
- Tier
- Billing Option / Pricing
- Number of instances

---

## 3）查 VM 时最容易错的地方

### 错误 1：忘记选操作系统
Linux 和 Windows 价格通常不同。

### 错误 2：区域选错
East US 和 Japan East 价格可能不一样。

### 错误 3：机型选错系列
D 系列和 F 系列价格与用途都不同。

### 错误 4：实例数量没改
明明客户要 2 台，结果只按 1 台算。

### 错误 5：把按量和 RI 混了
一定要明确你查的是哪种付费方式。

---

## 4）查 VM 后应该怎么记

建议在报价表里写成这样：

| 资源 | 区域 | 规格 | OS | 数量 | 计费方式 | 月费 |
| ---- | ---- | ---- | ---- | ---- | ---- | ---- |
| Web VM | East US | D4as v5 | Linux | 2 | Pay as you go | xxx |
| DB VM | East US | D8as v5 | Linux | 1 | 1-year RI | xxx |

---

# 七、查磁盘价格的基本方法

很多新手会漏掉这一项。

---

## 1）磁盘要分开看

通常至少分：

- 系统盘
- 数据盘

而且还要看：

- Standard SSD
- Premium SSD

---

## 2）查磁盘前要准备的信息

- 磁盘类型
- 容量
- 数量
- 区域（如适用）

例如：
- 64GB Standard SSD × 3
- 128GB Standard SSD × 2
- 512GB Premium SSD × 1

---

## 3）查磁盘时最容易错的地方

### 错误 1：只报 VM，不报盘
这是最常见问题。

### 错误 2：盘类型选错
数据库盘和应用盘通常性能要求不同。

### 错误 3：容量按错
尤其数据库项目，容量不能只按当前刚好够。

---

## 4）报价表里怎么记

| 资源 | 类型 | 容量 | 数量 | 月费 |
| ---- | ---- | ---- | ---- | ---- |
| OS Disk | Standard SSD | 64GB | 3 | xxx |
| Data Disk | Standard SSD | 128GB | 2 | xxx |
| DB Data Disk | Premium SSD | 512GB | 1 | xxx |

---

# 八、查 Blob Storage / 存储价格的基本方法

---

## 1）你要准备什么信息

- 预计容量
- 存储层级（Hot / Cool / Archive）
- 冗余方式（如适用）
- 是否有出网流量预估

---

## 2）常见场景

例如客户说：
- 存商品图片
- 存官网图片
- 存附件
- 存导出文件

你就可能需要查：
- Blob Storage
- 容量费用
- 请求 / 访问相关费用（视情况）
- 流量说明

---

## 3）查存储时最容易错的地方

### 错误 1：只查容量，不写访问和流量说明
很多时候存储本体不是唯一成本。

### 错误 2：层级选错
Hot / Cool / Archive 适用场景不同。

### 错误 3：不写预估前提
存储增长很常见，所以必须说明。

---

# 九、查 Redis 价格的基本方法

Redis 在电商和 APP 后台很常见。

---

## 1）你要准备什么信息

- 需要的 Redis 类型 / 层级
- 区域
- 是否生产环境
- 预估负载

---

## 2）查 Redis 时要注意什么

Redis 往往不是像 VM 那样只看 CPU / 内存，  
更多是按：

- 层级
- 容量
- 服务档位

来定价。

---

## 3）新手建议

如果客户没有非常明确的 Redis 规格要求，  
你可以先按：

- 基础版
- 标准版

做一版参考，再备注：
- 具体规格需结合并发和缓存量确认

---

# 十、查数据库价格的基本方法

这里要分两种情况。

---

# 十一、情况 A：数据库跑在 VM 上

如果数据库是装在 VM 上的，那数据库本身的软件许可暂且不论时，  
你主要查的是：

- VM
- 系统盘
- 数据盘
- 备份
- 网络

也就是说：

> **数据库 on VM，本质上先按 VM 项目报价。**

---

# 十二、情况 B：托管数据库服务

如果用的是：
- Azure SQL Database
- Azure Database for MySQL
- Azure Database for PostgreSQL

那你要查的重点就变成：

- 数据库计算规格
- 存储
- 备份保留
- 高可用（如果启用）

---

## 查托管数据库时最容易错的地方

### 错误 1：还按 VM 思路去想
托管数据库不一定有“系统盘”“数据盘”这种表达。

### 错误 2：忽略高可用选项
高可用常常会显著影响价格。

### 错误 3：忽略备份和存储增长
数据库费用不是只看“当前一个实例”。

---

# 十三、查网络类价格的基本方法

网络类通常容易被忽略，但实际项目里很重要。

---

## 常见要查的网络资源

- Public IP
- Load Balancer
- Application Gateway
- VPN Gateway
- CDN
- 出网流量

---

## 新手查网络类时的建议

### 1）固定资源和按量资源分开看
例如：
- Public IP、Load Balancer 可能偏固定结构
- 流量、CDN 流量偏用量

### 2）流量一定要单独备注
如果拿不到准数，也要说明：
- 暂未包含
- 或按预估值计算

---

# 十四、查 Backup 价格的基本方法

备份是很多人会漏，但又经常要补的。

---

## 1）你要准备什么信息

- 受保护实例数量
- 备份对象是什么（VM / 数据库）
- 预计备份容量
- 保留周期

---

## 2）查 Backup 时最容易错的地方

### 错误 1：只写“要备份”，不估容量
Backup 费用通常和数据量、保留周期有关。

### 错误 2：不写是 VM 备份还是数据库备份
不同对象，逻辑不一样。

### 错误 3：不写为估算值
备份费用很容易随着数据增长变化。

---

# 十五、中国版价格怎么查：思路和国际版不完全一样

中国版很重要的一点是：

> **你不能默认它和国际版计算器长得完全一样。**

很多时候你会更多依赖：

- Azure 中国官网对应产品价格页
- 指定服务的价格详情
- 国内价格政策页
- CPP 对应报价逻辑

---

## 中国版查价建议流程

### 第 1 步：确认中国版环境
- 区域：中国北部 / 中国东部
- 货币：人民币
- 优惠：CPP

### 第 2 步：进入 Azure 中国官网价格页
https://www.azure.cn/pricing/

### 第 3 步：按资源类型分别查
例如：
- VM
- 存储
- 数据库
- 网络
- 安全
- 备份

### 第 4 步：记录人民币价格
并区分：
- 按量
- CPP

---

## 中国版查价最容易错的地方

### 错误 1：拿国际版单价套中国版
这是最严重的问题之一。

### 错误 2：把 RI 写成中国版优惠
中国版通常更常见 CPP 表达。

### 错误 3：忽略中国版服务可用性差异
不是国际版有的东西，中国版都一样有。

---

# 十六、查价时最容易选错的 10 个地方

这是很重要的实操部分。

---

## 1. 版本选错
国际版和中国版搞混。

## 2. 区域选错
客户要 East US，你查成 Southeast Asia。

## 3. OS 选错
Linux 和 Windows 没区分。

## 4. VM 规格选错
D4as v5 和 D8as v5 选反。

## 5. 数量没改
客户要 2 台，你只算 1 台。

## 6. 盘漏了
只查了 VM，忘记查磁盘。

## 7. 盘类型选错
数据库盘本应 Premium SSD，却按 Standard SSD 算。

## 8. 按量和 RI / CPP 混了
列头和实际价格不一致。

## 9. 流量没写说明
导致客户后续看到账单有疑问。

## 10. 查完没及时记录
查了很多页面，最后数据散了，容易混乱。

---

# 十八、查完价格后，怎么整理成报价单

查完之后，不建议直接把计算器截图丢给销售或客户。

更好的方式是整理成结构化报价。

---

## 建议至少整理成 3 层

### 第一层：资源明细
逐项列出：
- 资源名
- 规格
- 数量
- 单价 / 小计

### 第二层：分类汇总
例如：
- 计算资源合计
- 磁盘资源合计
- 数据库 / 缓存合计
- 网络合计
- 安全与备份合计

### 第三层：总价与备注
例如：
- 月度总价
- 年度总价
- 按量版
- RI / CPP 版
- 备注说明

---

# 十九、一个最简单的查价实操例子

下面用一个很简化的例子帮你串一下。

---

## 场景
客户要：
- East US
- Linux Web VM D4as v5 × 2
- Linux DB VM D8as v5 × 1
- 64GB 系统盘 × 3
- 128GB 数据盘 × 2
- 512GB 数据库盘 × 1
- 1 个 Load Balancer
- 1 个 Blob Storage
- 1 套 Backup
- 给按量和 1 年 RI

---

## 你的动作应该是

### 1）先查 VM
- Web VM × 2
- DB VM × 1

### 2）再查 Disk
- 64GB Standard SSD × 3
- 128GB Standard SSD × 2
- 512GB Premium SSD × 1

### 3）再查网络与存储
- Load Balancer
- Blob Storage

### 4）再查 Backup
- 按当前实例和容量做估算

### 5）流量单独备注
- 如拿不到准数，写按实际结算或按预估值测算

### 6）整理成报价表
- 按量版一套
- RI 版一套
- 备注写清楚
