# NSG 规则测试

## 1. 是什么

NSG（Network Security Group，网络安全组）是 Azure 中用于控制网络流量访问规则的基础安全资源。

你可以先把它理解成：

> 一组决定“谁可以访问谁、访问哪个端口、是允许还是拒绝”的网络规则。

NSG 常用于控制这些场景：

- 谁可以访问一台 VM
- 哪些端口可以从公网进入
- 哪些子网之间可以互通
- 某类流量是否应该被拒绝

在 Azure 学习早期，NSG 是非常关键的概念，因为很多“连不上”的问题，本质上都和它有关。

---

## 2. 解决什么问题

NSG 主要解决下面几类问题：

### 2.1 控制入站访问
例如：

- 是否允许公网访问 Windows VM 的 3389 端口
- 是否允许公网访问 Linux VM 的 22 端口
- 是否允许某个管理 IP 访问某台服务器

### 2.2 控制出站访问
例如：

- 某台 VM 是否允许访问互联网
- 是否允许访问某个内部服务
- 是否限制特定方向的网络连接

### 2.3 限制不必要的暴露面
不是所有端口都应该开放。  
NSG 可以帮助你只开放真正需要的端口，降低风险。

### 2.4 做最基础的网络隔离
即使资源在同一个 VNet 中，也可以通过 NSG 对不同流量进行控制。

---

## 3. NSG 最核心的作用是什么

可以先记住一句话：

> NSG 决定某段网络流量是允许通过，还是拒绝通过。

它的判断通常围绕这些条件：

- 方向：入站 / 出站
- 来源：谁发起访问
- 目标：访问谁
- 协议：TCP / UDP / Any
- 端口：访问哪个端口
- 动作：Allow / Deny
- 优先级：哪条规则先匹配

所以你以后看到 NSG，不要只把它理解成“安全设置”，而要理解成：

> 一个网络流量匹配规则系统。

---

## 4. NSG 可以绑定到哪里

NSG 常见可以关联到两个层级：

### 4.1 子网（Subnet）
如果把 NSG 绑定到子网，那么这组规则会影响该子网中的资源流量。

### 4.2 网卡（NIC）
如果把 NSG 绑定到某台 VM 的网卡，那么这组规则会更直接地影响该 VM 的网络访问。

---

## 5. 学习阶段先怎么理解

学习阶段你可以先用最简单的方式理解：

- **VM 能不能被连上**
- 很多时候取决于 **NSG 有没有放通对应端口**

例如：

- Windows VM 的 RDP 默认常看 **3389**
- Linux VM 的 SSH 默认常看 **22**

如果 NSG 不允许这些端口，即使：

- VM 是 Running
- Public IP 也正常
- 用户名密码也对

你依然可能连不上。

---

## 6. NSG 规则的基本组成

一条 NSG 规则通常会包含这些字段。

### 6.1 Priority（优先级）
数字越小，优先级越高。  
Azure 会优先匹配优先级更高的规则。

例如：

- 100 比 200 更先匹配
- 如果前面的规则已经匹配成功，后面的规则通常不会再继续判断

### 6.2 Source（来源）
表示流量从哪里来，例如：

- Any
- IP Addresses
- Service Tag
- VirtualNetwork
- Internet

### 6.3 Source port ranges（来源端口）
很多场景可以保持默认 `*`。

### 6.4 Destination（目标）
表示流量要去哪里，例如：

- Any
- IP Addresses
- VirtualNetwork

### 6.5 Destination port ranges（目标端口）
例如：

- 22
- 80
- 443
- 3389

### 6.6 Protocol（协议）
例如：

- TCP
- UDP
- Any

### 6.7 Action（动作）
最常见的是：

- Allow
- Deny

### 6.8 Direction（方向）
分成：

- Inbound（入站）
- Outbound（出站）

---

## 7. 入站和出站怎么理解

### 7.1 入站（Inbound）
表示外部流量进入资源。

例如：

- 你的电脑远程桌面连接 Azure Windows VM
- 你的终端 SSH 到 Azure Linux VM
- 浏览器访问某台 Web 服务器的 80 / 443

这些都属于入站流量。

---

### 7.2 出站（Outbound）
表示资源主动访问外部。

例如：

- VM 访问互联网下载软件
- VM 访问数据库
- VM 访问更新源
- VM 调用某个外部 API

这些属于出站流量。

---

## 8. 怎么测试

学习 NSG 最好的方式不是背定义，而是做几个最小实验。

---

### 8.1 测试 1：放通 RDP 3389

目标：

- 创建一台 Windows VM
- 在 NSG 中允许 TCP 3389
- 用 RDP 测试能否连接

观察点：

- 规则加上后是否能连接
- 如果删掉规则，RDP 是否失败

---

### 8.2 测试 2：放通 SSH 22

目标：

- 创建一台 Linux VM
- 在 NSG 中允许 TCP 22
- 用 SSH 测试能否连接

观察点：

- 规则加上后是否能 SSH 登录
- 移除规则后是否连接失败

---

### 8.3 测试 3：只允许自己的公网 IP 访问

目标：

- 不再把来源写成 `Any`
- 只允许你当前公网 IP 访问 22 或 3389

观察点：

- 是否仍然能正常连接
- 如果换网络后是否失效

这个实验可以帮助你理解：

> NSG 不只是“开或关”，而是可以控制“谁能访问”。

---

### 8.4 测试 4：放通 Web 端口 80 / 443

目标：

- 在 Linux VM 中安装 Nginx 或 Apache
- NSG 放通 80 或 443
- 用浏览器访问 Public IP

观察点：

- 如果 NSG 不开，浏览器通常打不开
- 如果 NSG 放通但服务没启动，也打不开

这个实验可以帮助你理解：

> 网络放通只是条件之一，应用服务本身也要正常。

---

## 9. 一个最简单的理解例子

假设你有一台 Linux VM，Public IP 是存在的，VM 状态也是 Running。

现在你执行：

```bash
ssh azureuser@<Public-IP>
```

如果 NSG 中没有允许 TCP 22，那么大概率会出现：

- 连接超时
- 无法建立连接

但如果你加上一条入站允许规则：

- Source：Any（或你的公网 IP）
- Protocol：TCP
- Destination Port：22
- Action：Allow
- Direction：Inbound

那么 SSH 才有机会成功。

也就是说：

> NSG 决定的是“这条访问请求有没有资格进入”。

## 10. 常见问题

### 问题 1：NSG 和防火墙有什么区别？

学习阶段可以先简单理解为：

- **NSG**：Azure 网络层面的访问控制
- **系统防火墙**：操作系统内部的访问控制

也就是说，即使 NSG 放通了端口，系统内部防火墙或服务没开，访问仍可能失败。

---

### 问题 2：为什么我已经有 Public IP 还是访问不了？

因为有 Public IP 只说明“地址有了”，不说明“流量一定放行”。

还要继续检查：

- NSG 是否放通对应端口
- 操作系统是否允许
- 对应服务是否真的在监听端口
- 用户名 / 密码 / SSH Key 是否正确

---

### 问题 3：为什么规则写了还是不生效？

常见原因包括：

- 规则绑错了 NSG
- 规则方向选错了
- 端口写错了
- 协议写错了
- 优先级被前面的规则覆盖
- 来源 IP 不匹配
- 实际生效的是子网或 NIC 上另一套规则

---

### 问题 4：优先级怎么理解？

可以先理解成：

> 数字越小，越先匹配。

比如：

- 100 Allow 22
- 200 Deny Any

那访问 22 时，通常会先命中优先级 100 的 Allow。

但如果前面先有一条更高优先级 Deny，也可能导致后面的 Allow 没机会生效。

---

### 问题 5：学习阶段要不要直接开 Any？

学习测试阶段可以适度使用 `Any` 来快速验证现象。  
但要明确知道：

- 这只是为了学习方便
- 不是正式环境推荐做法

正式环境更建议：

- 限制来源 IP
- 最小化开放端口
- 只开放真正必要的访问

---

## 11. 排查 NSG 问题时怎么想

遇到网络访问失败时，可以按下面顺序想：

### 11.1 我访问的是什么协议和端口
例如：

- RDP → 3389 / TCP
- SSH → 22 / TCP
- HTTP → 80 / TCP
- HTTPS → 443 / TCP

### 11.2 这是入站还是出站
如果是“我从本机连 Azure VM”，通常重点先看入站规则。

### 11.3 来源是谁
是：

- Any
- Internet
- 你的固定公网 IP
- 某个网段

### 11.4 规则绑定在哪
是绑定在：

- 子网
- 还是 NIC

### 11.5 有没有别的规则先拦截了
例如：

- 更小优先级的 Deny
- 其他更早命中的规则

---

## 12. 一个基础排查清单

如果你怀疑问题和 NSG 有关，可以先快速检查下面这些项。

### 12.1 资源层
- VM 是否 Running
- Public IP 是否存在
- 是否访问的是正确 IP

### 12.2 NSG 绑定层
- VM 网卡是否绑定 NSG
- 子网是否绑定 NSG
- 实际生效的是哪一层规则

### 12.3 规则层
- 是 Inbound 还是 Outbound
- 协议是否正确
- 端口是否正确
- 来源是否正确
- 优先级是否合理
- 动作是否为 Allow

### 12.4 系统层
- 对应服务是否启动
- 系统防火墙是否允许
- 用户名 / 密码 / SSH Key 是否正确

---

## 13. 客户常问的问题

这一部分重点不是只记录客户怎么问，而是练习你该怎么回答。

---

### 问题 1：NSG 是干什么的？

**简短回答：**  
NSG 是 Azure 里用来控制网络流量访问规则的基础安全组件。

**展开回答：**  
它决定哪些流量可以进入资源、哪些流量可以出去。  
比如客户最常见会遇到的就是：

- Windows VM 的 3389 要不要开放
- Linux VM 的 22 要不要开放
- 网站的 80 / 443 要不要开放
- 是不是所有人都能访问，还是只允许指定 IP

所以 NSG 本质上是在控制“谁能访问什么”。

---

### 问题 2：为什么我明明有 VM 和公网 IP，还是连不上？

**简短回答：**  
因为是否能访问，不只取决于 IP 是否存在，还要看 NSG 有没有允许对应端口。

**展开回答：**  
很多客户会以为“有公网 IP 就一定能连”，其实不是。  
公网 IP 只是地址，真正能不能访问还要继续看：

- NSG 是否放通
- 系统防火墙是否允许
- 服务是否启动
- 用户名和认证是否正确

---

### 问题 3：正式环境为什么不建议把端口都开给 Any？

**简短回答：**  
因为这样暴露面太大，风险更高。

**展开回答：**  
学习环境为了快速验证，可以临时开放。  
但正式环境通常更建议：

- 只开放必要端口
- 只允许固定来源 IP
- 配合 VPN / Bastion / 跳板机
- 做更细的访问控制和审计

所以 NSG 的价值不只是“放通”，更重要的是“限制不必要访问”。

---

### 问题 4：NSG 和系统防火墙是不是一回事？

**简短回答：**  
不是。NSG 是 Azure 网络层的规则，系统防火墙是操作系统内部规则。

**展开回答：**  
你可以理解为：

- NSG 决定流量能不能先到达这台机器
- 系统防火墙决定流量到了机器后，系统接不接受

所以两层都可能影响最终访问结果。

---

## 14. 你可以怎么对客户解释

如果客户第一次接触 NSG，不建议一上来讲很多字段名。  
可以先这样说：

> NSG 可以理解为 Azure 里的网络访问规则表。  
> 它决定哪些流量能进入资源、哪些流量不能进入。  
> 比如一台服务器要不要开放 22、3389、80、443 这些端口，通常都会通过 NSG 来控制。  
> 所以 NSG 的核心价值不是单纯“开端口”，而是按需要允许、按风险限制访问。

## 17. 本篇总结

NSG 是 Azure 中非常基础、也非常高频的网络安全控制组件。

学习这一步时，真正重要的不只是记住字段，而是理解：

- NSG 控制的是网络流量能不能通过
- 访问失败时，要先分清是网络层问题还是系统层问题
- RDP、SSH、HTTP、HTTPS 等常见访问都和 NSG 有直接关系
- 学习测试可以先用简单规则验证现象，但正式环境要更强调最小暴露面

可以先记住一句话：

> **NSG 决定流量能不能进，系统和服务决定进来以后能不能真正用。**
