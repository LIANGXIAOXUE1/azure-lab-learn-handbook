# 报价案例：数据库上云

这篇文档的目标是：

> **通过“数据库上云”案例，帮助你理解数据库类报价为什么要比普通应用服务器更谨慎。**

很多新手在做 Azure 报价时，容易把数据库理解成：

- 一台服务器
- 装个 MySQL / SQL Server
- 给个规格
- 就结束了

但实际工作里，数据库上云往往比 Web / 应用服务器更敏感，因为它更关心：

- CPU
- 内存
- 磁盘性能
- 数据容量
- 备份
- 高可用
- 恢复能力
- 网络访问控制

所以这一篇会重点帮助你理解：

1. 数据库上云常见有哪些报价思路  
2. 为什么数据库不能总按应用服务器的思路去报  
3. 为什么磁盘、备份、高可用在数据库项目里特别重要  
4. VM 上数据库和托管数据库，在报价思路上有什么区别  
5. 数据库上云项目里最容易漏哪些项  

---

## 当前诉求
先做数据库上云预算，核心诉求包括：

- 把数据库迁到 Azure
- 保证数据安全
- 要有基础备份能力
- 最好能考虑后续扩展
- 需要生产环境方案

## 部署区域
- 中国北部 3

## 结算方式偏好
- 人民币
- 希望看按量和 CPP 参考

- 数据库类型

# 四、第二步：继续补需求

数据库上云项目里，这一步非常关键。

---

## 至少要继续确认这些问题

### 1）数据库类型是什么
- MySQL
- SQL Server
- PostgreSQL
- MariaDB
- 其他

---

## 2）数据库部署方式想怎么做
是：
- 跑在 VM 上
还是
- 用托管数据库服务

这个问题会直接决定后面报价结构完全不同。

---

## 3）当前数据库版本
例如：
- MySQL 5.7 / 8.0
- SQL Server 2019
- PostgreSQL 14

为什么要问？
因为不同版本会影响：
- 迁移方案
- 兼容性
- 服务可选范围

---

## 4）当前数据量多大
例如：
- 100GB
- 500GB
- 2TB

为什么重要？
因为这会影响：
- 磁盘容量
- 存储费用
- 备份费用
- 增长空间评估

---

## 5）读写压力大概如何
例如：
- 一般业务库
- 读多写少
- 写入频繁
- 高峰明显

为什么重要？
因为数据库选型不只是看“数据量”，还要看“压力”。

---

## 6）是否要求高可用
这一定要问清楚。

例如客户可能要求：
- 可以接受单机
- 需要主备
- 需要高可用
- 不能长时间中断

这会显著影响成本。

---

## 7）备份策略
要继续问：
- 备份保留几天
- 是否需要长期保留
- 是否有误删恢复需求
- 是否要求跨区备份

---

## 8）网络访问方式
数据库是给谁访问的？

例如：
- 云上应用访问
- 本地机房访问
- VPN / 专线打通
- 仅内网开放
- 是否允许公网访问（一般不建议）

---

## 9）操作系统
如果是 VM 方案，还要问：
- Linux
- Windows

尤其如果是 SQL Server on VM，系统与授权都会影响价格。

---

## 10）客户后续是否还会迁应用
这个问题虽然不直接影响数据库单价，但会影响你是否需要提前考虑：
- 网络互通
- 架构扩展
- 后续整体云上布局

# 六、先建立一个重要概念：数据库上云常见有两种思路

数据库上云，最常见的思路通常有两类：

---

## 方案 A：数据库跑在 VM 上

也就是：

> **先买一台云服务器，再在上面安装 MySQL / SQL Server / PostgreSQL。**

你可以理解成：

- 更像“把本地数据库服务器搬到云上”
- 管理自由度更高
- 也更像传统运维方式

---

## 方案 B：使用托管数据库服务

也就是：

> **直接购买 Azure 提供的数据库服务，而不是自己管整台服务器。**

你可以理解成：

- 数据库本身交给云平台托管更多
- 不再自己重点管操作系统
- 更偏“买服务”而不是“买服务器”

---

# 七、这两种思路的核心区别

这个区别一定要吃透。

---

## VM 上数据库

你负责更多内容：
- VM 规格
- 系统
- 磁盘
- 数据库安装
- 运维
- 补丁
- 性能调优
- 备份策略（部分）
- 高可用方案（如需）

### 报价时通常要报
- VM
- 系统盘
- 数据盘
- 备份
- 网络访问
- 安全
- 监控（按需）

---

## 托管数据库

平台帮你承担更多底层工作：
- 部分运维
- 基础高可用能力（视服务而定）
- 自动备份（视 SKU 与策略而定）
- 平台级管理

### 报价时通常重点看
- 数据库规格
- 存储容量
- 备份保留
- 高可用选项
- 网络访问方式

---

# 八、这篇案例就按“两套方案对比”来做

这样最适合练习，也最符合实际项目沟通方式。

---

# 九、方案 A：MySQL 跑在 VM 上

先看第一种：客户买一台 VM，自己在上面跑 MySQL。

---

## 1）系统结构怎么想

### 计算层
- 1 台数据库 VM

### 磁盘层
- 系统盘
- 数据盘

### 数据层
- MySQL 软件部署在 VM 上

### 网络层
- 数据库只允许业务内网访问
- 可能涉及 VNet / 内网策略 / VPN

### 安全与运维层
- Backup
- Defender
- 监控（按需）

---

## 2）资源拆解示例

### 计算资源
- Linux VM D8as v5 × 1

### 磁盘资源
- 系统盘：64GB Standard SSD × 1
- 数据盘：512GB Premium SSD × 1  
  （因为当前数据量约 400GB，还要考虑增长和性能）

### 安全与备份
- Backup × 1
- Defender × 1（按适用策略）

### 网络
- 内网访问相关资源（按实际网络架构决定是否单独列）
- 如有 VPN / ExpressRoute，需另行评估

---

## 3）为什么数据库 VM 用 D8as v5

因为这是生产环境，数据量约 400GB，而且是业务数据库。

D8as v5 大致是：
- 8 vCPU
- 32GB 内存

这类规格通常比 D4as v5 更适合数据库，因为数据库更依赖：

- CPU
- 内存
- 稳定性

尤其生产库，一般不建议轻易从太低规格起步。

---

## 4）为什么数据盘要用 Premium SSD

数据库和普通文件盘不一样。  
数据库的磁盘更敏感，因为会涉及：

- 表读写
- 索引访问
- 事务提交
- 日志写入
- 高峰查询

如果磁盘性能太一般，常见问题会是：

- 查询慢
- 响应抖动
- 高峰期卡顿
- 业务系统整体变慢

所以数据库盘通常比应用服务器更应该重视性能。

---

## 5）为什么数据盘不能只按当前 400GB 配 400GB

因为数据库容量不是静态的。

你至少要考虑：
- 当前已有数据
- 未来增长
- 备份空间影响
- 运维预留空间

所以实践里通常不会“刚好配满”。

---

# 十、方案 A 的优点和不足

---

## 优点

### 1）更接近客户原有本地部署习惯
客户如果原来就是自己维护数据库服务器，接受度通常较高。

### 2）自由度更高
- 可以自己控制 OS
- 可以自己装软件
- 可以自己调参数

### 3）有些老系统更容易平移
特别是迁移老数据库系统时，这种方式更容易让客户感觉“像搬家”。

---

## 不足

### 1）需要自己管更多
包括：
- OS
- 数据库安装
- 运维
- 打补丁
- 备份策略
- 安全

### 2）高可用要自己进一步设计
如果客户后续要求主备、集群、高可用，这部分复杂度会明显上升。

### 3）更容易漏项
因为资源分散在：
- VM
- Disk
- Backup
- 安全
- 网络
里，新手很容易漏。

---

# 十一、方案 B：使用托管 MySQL 服务

再看第二种：不自己管整台数据库服务器，而是直接用 Azure 托管数据库。

---

## 1）系统结构怎么想

### 数据库服务层
- 托管 MySQL 实例

### 存储层
- 服务自带或按配置分配存储

### 网络层
- 内网访问控制
- 私网接入 / 防火墙规则（按实际支持方式）

### 备份与高可用层
- 看所选服务级别
- 备份保留策略
- 高可用选项

---

## 2）资源拆解思路示例

这里不一定非要细到某个 SKU 名字，新手阶段你可以先按下面思路理解：

### 托管数据库资源
- Azure Database for MySQL × 1

### 需要重点确认的参数
- 计算规格
- 存储容量
- 是否高可用
- 备份保留期
- 网络接入方式

### 可能还会涉及
- 监控
- 日志
- 出口 / 网络策略

---

## 3）托管数据库报价时，重点关注什么

和 VM 方案不一样，托管方案你通常不重点报：

- OS
- 系统盘
- 数据盘（以 VM 盘的方式）

而是更关注：

- 数据库本身的性能层级
- vCore / 计算能力
- 存储大小
- 自动备份策略
- 高可用是否启用

---

# 十二、方案 B 的优点和不足

---

## 优点

### 1）平台托管更多底层能力
客户不用自己重点管：
- 操作系统
- 很多基础维护工作

### 2）备份和管理通常更标准化
对于希望少管运维的客户，这通常更友好。

### 3）更适合云原生思路
如果客户未来应用也会上云，托管数据库通常更容易融入整体云架构。

---

## 不足

### 1）自由度可能不如 VM 高
有些客户会觉得：
- 不能像自建数据库服务器那样完全控制环境

### 2）老系统兼容性要确认
并不是所有原有数据库用法都适合直接平移到托管服务。

### 3）价格结构对新手来说不如 VM 直观
因为新手更容易理解“服务器 + 盘”，而托管数据库更像一个整合服务。

---

# 十三、两种方案怎么对比理解

下面给你一个新手最容易记住的对比方式。

| 对比点 | VM 上数据库 | 托管数据库 |
| ---- | ---- | ---- |
| 报价思路 | 报服务器 + 磁盘 + 备份 | 报数据库服务本身 + 存储 + 高可用 |
| 管理方式 | 更像自己养一台数据库服务器 | 更像直接买数据库服务 |
| 自由度 | 更高 | 相对更规范 |
| 运维工作量 | 更大 | 更低 |
| 老系统迁移 | 通常更容易平移 | 需确认兼容性 |
| 新手易漏项 | 更容易漏盘、备份、安全 | 更容易漏高可用、备份保留、网络访问 |

---

# 十四、这个案例里，为什么建议两套都给客户看

因为很多客户在数据库上云时，本身就处于选择阶段：

- 是继续按原思路搬到云上
- 还是顺便转成托管服务

如果你能同时给两套思路，客户会更容易比较：

- 成本
- 管理复杂度
- 可维护性
- 后续扩展性

这通常比只给单一方案更专业。

---

# 十五、数据库上云项目里，哪些是主费用项

这点一定要有感觉。

---

## 对 VM 方案来说，主费用通常是：

### 1）数据库 VM
例如 D8as v5

### 2）数据库数据盘
尤其如果是 Premium SSD，成本通常不低

### 3）备份
随着数据量增长，备份费用会越来越明显

### 4）高可用（如果后续加）
一旦从单机变成主备，成本会明显上升

---

## 对托管方案来说，主费用通常是：

### 1）数据库计算规格
### 2）数据库存储
### 3）高可用选项
### 4）备份保留与相关能力

---

# 十六、数据库项目里最容易漏什么

这是新手非常容易出错的部分。

---

## 1）数据盘
很多人只报 VM，不报数据盘。

对于数据库来说，这几乎是最典型的漏项之一。

---

## 2）磁盘性能等级
即使报了数据盘，也容易忘了区分：
- Standard SSD
- Premium SSD

但对数据库来说，这个差异很关键。

---

## 3）备份
客户说“上云”，你不能只考虑“怎么跑”，还要考虑“怎么恢复”。

---

## 4）高可用
客户一开始可能没主动提，但生产数据库最好至少问一句。

---

## 5）网络访问限制
数据库一般不建议随意公网开放。  
至少要确认：
- 谁访问它
- 走什么网络
- 是否需要 VPN / 专线 / 私网

---

## 6）存储增长
数据库不会一直停在当前容量。  
如果现在 400GB，你不能完全按“400GB 刚好够”去想。

---

## 7）监控与日志
数据库一旦出问题，性能和日志分析很重要。  
虽然不一定每次都单独报，但至少要知道是否需要。

---

# 二十、如果客户预算敏感，怎么做更合理

数据库项目里，预算敏感很常见，但降配要比 Web 更谨慎。

---

## 可以考虑的方式

### 1）先保持数据库规格不轻易大降
因为数据库一旦性能不足，影响的是整套业务。

### 2）先不做高可用增强项
如果客户只是先做预算，可以：
- 基础生产版先单实例
- 高可用作为增强版单列

### 3）备份策略做分层说明
例如：
- 基础备份版
- 更长保留期版

### 4）做 VM 版和托管版对比
有时不一定是谁绝对便宜，而是谁更适合客户现阶段。

# 二十一、这个案例真正想教你的方法

而是建立一个数据库报价思维。

- 部署方式（VM / 托管）
- 数据量
- 备份要求

- 基础版

# 二十四、这篇你要记住的核心结论

## 1. 数据库上云不是只报一台服务器
数据库项目通常还要重点考虑盘、备份、高可用和网络访问。

## 2. 数据库比普通应用服务器更吃内存和磁盘性能
所以通常不建议完全按应用层思路去配。

## 3. 数据库上云常见有两种思路
- VM 自建数据库
- 托管数据库服务

## 4. 两种方案的报价逻辑不同
VM 方案更像“报服务器 + 盘 + 备份”，托管方案更像“报数据库服务能力本身”。

## 5. 数据库项目最容易漏的项包括
- 数据盘
- 磁盘性能
- 备份
- 高可用
- 网络访问控制
- 存储增长
