# 远程连接测试：RDP 与 SSH

## 1. 是什么

在 Azure 中，创建完虚拟机（VM）之后，通常不会只停留在“资源创建成功”这一步。  
更重要的是确认：

> 你能不能真正连上这台机器，并且知道连接失败时该从哪里开始排查。

对于最常见的两类虚拟机，远程连接方式通常是：

- **Windows VM** → 使用 **RDP（Remote Desktop Protocol）**
- **Linux VM** → 使用 **SSH（Secure Shell）**

所以这篇文档的重点是：

- 理解 RDP 和 SSH 分别是什么
- 知道如何做最基础的连接测试
- 知道连接失败时优先检查哪些层

---

## 2. 解决什么问题

远程连接测试主要解决的是下面这些问题：

### 2.1 验证虚拟机是不是“真的可用”
很多时候 Azure Portal 显示 VM 已经创建成功，但这不代表你一定能正常访问它。  
只有真正连上去，才能说明：

- VM 已启动
- 网络路径基本通
- 账号认证有效
- 对应端口已正确放行

### 2.2 验证网络与安全配置是否正确
无论是 RDP 还是 SSH，本质上都依赖：

- Public IP 或可达的私网地址
- NSG 规则
- 操作系统本身的远程访问能力
- 正确的账号和认证方式

所以远程连接测试，是验证“计算 + 网络 + 安全”是否协同正常的最好入口。

### 2.3 为后续部署和运维打基础
只有能成功远程连接，后面你才能继续做这些事：

- 安装软件
- 修改配置
- 上传文件
- 运行命令
- 做排障
- 观察系统状态

---

## 3. RDP 和 SSH 分别是什么

### 3.1 RDP 是什么

RDP（Remote Desktop Protocol）是微软常用的远程桌面协议。  
它主要用于远程连接 Windows 系统。

你可以把它简单理解为：

> 在本地电脑上远程打开一台云上的 Windows 桌面。

默认常见端口是：

- **TCP 3389**

---

### 3.2 SSH 是什么

SSH（Secure Shell）是一种常见的远程命令行连接协议，主要用于 Linux 系统。

你可以把它简单理解为：

> 在本地终端里远程登录一台 Linux 服务器，并执行命令。

默认常见端口是：

- **TCP 22**

---

### 3.3 它们的核心区别

最基础的理解方式可以是：

- **RDP**：远程桌面，主要面向 Windows，图形界面操作为主
- **SSH**：远程命令行，主要面向 Linux，终端命令操作为主

当然这只是入门层面的理解，实际环境里会更复杂，但学习阶段先这样记就够用了。

---

## 4. 远程连接前要先检查什么

不管是 RDP 还是 SSH，正式连接前都建议先确认下面几个条件。

### 4.1 VM 是否处于 Running 状态
如果虚拟机还没有启动完成，或者已经停止，那么你当然无法连接。

### 4.2 是否有可访问的 IP
常见情况有两种：

- 有 Public IP，可以从公网直接连接
- 没有 Public IP，需要通过内网、跳板机、VPN 或 Bastion 等方式连接

学习阶段最常见的是直接给测试 VM 分配 Public IP。

### 4.3 对应端口是否已放通
常见端口包括：

- Windows RDP：TCP 3389
- Linux SSH：TCP 22

如果 NSG 没有允许这些端口，连接通常会失败。

### 4.4 用户名和认证方式是否正确
例如：

- Windows VM：用户名 / 密码
- Linux VM：用户名 / SSH Key 或密码

### 4.5 本地网络是否允许连接
有些企业网络、学校网络或公共网络环境可能限制某些端口或协议。

---

## 5. RDP 测试步骤（Windows VM）

### 5.1 获取 Windows VM 的 Public IP

进入 Azure Portal：

1. 打开目标 Windows VM
2. 查看 `Overview`
3. 记录 `Public IP address`

---

### 5.2 检查 3389 是否放通

进入：

- VM → `Networking`

确认：

- 入站规则中允许 `RDP`
- 或者允许 `TCP 3389`

---

### 5.3 打开远程桌面连接工具

在 Windows 本机上：

1. 打开“远程桌面连接”
2. 输入 VM 的 Public IP
3. 点击连接

---

### 5.4 输入账号和密码

输入你创建 VM 时设置的：

- Administrator username
- Password

如果认证通过，通常就能进入远程 Windows 桌面。

---

### 5.5 登录成功后建议做的几个动作

如果登录成功，建议立刻记录并验证：

- 当前用户名
- 当前主机名
- 系统版本
- 网络连通性
- 是否可正常打开服务器管理器 / PowerShell

这样可以确认这台 VM 不只是“连上了”，而是真的可用。

---

## 6. SSH 测试步骤（Linux VM）

### 6.1 获取 Linux VM 的 Public IP

进入 Azure Portal：

1. 打开目标 Linux VM
2. 查看 `Overview`
3. 记录 `Public IP address`

---

### 6.2 检查 22 端口是否放通

进入：

- VM → `Networking`

确认：

- 入站规则中允许 `SSH`
- 或者允许 `TCP 22`

---

### 6.3 在本地终端发起 SSH 连接

例如：

```bash
ssh azureuser@<Public-IP>
```

如果你使用的是私钥文件，命令可能类似：

```bash
ssh -i ~/.ssh/id_rsa azureuser@<Public-IP>
```

---

### 6.4 首次连接时确认主机指纹

首次连接时，通常会看到类似提示。  
输入：

```bash
yes
```

然后继续。

---

### 6.5 登录成功后建议做的几个动作

如果登录成功，建议先执行几个基础命令：

```bash
whoami
hostname
pwd
ls
```

如果这些命令都正常，说明 SSH 连接和基础系统状态大概率没有问题。

---

## 7. 怎么测试

这一篇建议至少做下面几组测试。

### 7.1 Windows VM：RDP 成功测试

验证点：

- VM 为 Running
- 有 Public IP
- 3389 已放通
- 用户名密码正确
- 能成功进入桌面

---

### 7.2 Linux VM：SSH 成功测试

验证点：

- VM 为 Running
- 有 Public IP
- 22 已放通
- 用户名正确
- 私钥或密码正确
- 能成功进入终端

---

### 7.3 失败场景模拟测试

建议你至少故意观察一次失败场景，例如：

- 不放通 22 或 3389
- 用户名输错
- 记录错误 IP
- 使用错误私钥

这样你会更清楚连接失败时的现象和排查入口。

---

## 8. 远程连接失败时先查哪几层

远程连接失败时，不建议一上来就乱点。  
更有效的方法是按层排查。

---

### 8.1 第一层：资源状态层

先看：

- VM 是否真的创建成功
- VM 是否是 Running
- VM 是否分配了正确的 IP
- Public IP 是否还存在

如果资源本身状态不对，后面都不用查。

---

### 8.2 第二层：网络路径层

再看：

- 目标 IP 是否正确
- 本地网络是否能访问外网对应端口
- 是否存在公司网络 / 防火墙限制
- 是否确实需要公网访问，还是应该走内网方式

---

### 8.3 第三层：NSG / 端口层

重点看：

- 3389 是否允许（Windows）
- 22 是否允许（Linux）
- 来源 IP 是否被限制
- 是否有更高层规则覆盖或阻断

---

### 8.4 第四层：账号与认证层

如果网络是通的，但还是进不去，要看：

- 用户名是否正确
- 密码是否正确
- SSH Key 是否匹配
- 是否用了错误的私钥文件

---

### 8.5 第五层：系统内部层

如果前面都没问题，还可能是系统本身问题，例如：

- RDP 服务异常
- SSH 服务异常
- Windows 防火墙限制
- Linux SSH 配置异常
- 系统启动未完成
- 系统负载异常

---

## 9. 常见问题

### 问题 1：VM 显示 Running，为什么还是连不上？

因为 Running 只说明虚拟机在运行，不代表远程访问路径一定正常。  
还要继续检查：

- Public IP
- NSG
- 用户名密码 / SSH Key
- 本地网络
- 系统服务

---

### 问题 2：RDP 连不上时最先看什么？

建议先看这几个：

1. VM 是否有 Public IP
2. NSG 是否允许 TCP 3389
3. 用户名密码是否正确
4. 本机网络是否限制 RDP
5. VM 是否真的完成启动

---

### 问题 3：SSH 连不上时最先看什么？

建议先看这几个：

1. VM 是否有 Public IP
2. NSG 是否允许 TCP 22
3. 用户名是否正确
4. SSH Key 是否匹配
5. 本地终端命令是否写对

---

### 问题 4：为什么会出现 Connection timed out？

这通常更像“网络层不通”，常见原因包括：

- IP 不对
- 端口没放通
- NSG 拦截
- 本地网络限制
- 目标 VM 没真正对外可达

---

### 问题 5：为什么会出现 Authentication failed / Permission denied？

这通常更像“认证层失败”，常见原因包括：

- 用户名不对
- 密码错误
- SSH 私钥错误
- SSH 公钥没有正确配置
- 认证方式和创建时不一致

---

## 10. 客户常问的问题

这一部分重点不是只记录客户怎么问，而是练习你该怎么回答。

---

### 问题 1：创建完 VM 后为什么还要做远程连接测试？

**简短回答：**  
因为资源创建成功不等于真正可访问，远程连接测试是验证 VM 是否真正可用的关键一步。

**展开回答：**  
很多时候 Portal 显示部署成功，但如果网络、端口、认证方式有问题，用户仍然无法登录。  
所以远程连接测试本质上是在验证：

- 机器起来了没有
- 网络通不通
- 端口开没开
- 账号能不能用

---

### 问题 2：Windows 和 Linux 的远程连接方式有什么区别？

**简短回答：**  
Windows 常用 RDP，Linux 常用 SSH。

**展开回答：**  
Windows 更常见的是图形化远程桌面管理，所以用 RDP；  
Linux 更常见的是命令行方式管理，所以用 SSH。  
两者背后的排查逻辑类似，但协议、端口和认证方式不同。

---

### 问题 3：为什么我明明创建了 Public IP 还是连不上？

**简短回答：**  
因为能不能连接，不只取决于有没有 Public IP，还要看端口、NSG、认证方式和系统状态。

**展开回答：**  
Public IP 只是“地址存在了”，但如果：

- 3389 / 22 没开
- 用户名密码不对
- SSH Key 不匹配
- 系统服务异常

还是无法连接。

---

### 问题 4：为什么正式环境不建议直接开放公网 RDP / SSH？

**简短回答：**  
因为直接把管理端口暴露到公网，安全风险会更高。

**展开回答：**  
学习环境为了方便测试，可以直接放通。  
但正式环境通常会更谨慎，例如：

- 限制来源 IP
- 通过 VPN / Bastion
- 使用跳板机
- 做更严格的身份验证和审计

所以学习阶段和生产阶段的做法会有差别。

---

## 11. 你可以怎么对客户解释

如果客户第一次接触这部分，不建议一上来就讲很多协议细节。  
可以先从验证价值来解释。

你可以这样说：

> 创建完一台 VM，不代表它已经真的能用。  
> 对 Windows 来说，通常要进一步测试能不能通过远程桌面登录；对 Linux 来说，通常要测试能不能通过 SSH 登录。  
> 这一步的意义是验证机器本身、网络路径、端口放通和账号认证是不是都正常。  
> 如果这一步不做，后面即使资源看起来已经部署成功，也可能还是无法真正使用。

---

## 12. 实践建议

学习完这一篇后，建议至少做下面几个动作：

### 12.1 对 Windows VM 做一次完整 RDP 测试
记录：

- Public IP
- 用户名
- 是否成功登录
- 失败时的报错信息

### 12.2 对 Linux VM 做一次完整 SSH 测试
记录：

- Public IP
- 用户名
- SSH 命令
- 是否成功登录
- 失败时的报错信息

### 12.3 故意观察一次失败现象
比如：

- 把端口关掉再测试
- 故意输错用户名
- 故意使用错误私钥

这样你会更容易建立排障经验。

### 12.4 记录自己的排查顺序
例如：

1. 先看 VM 是否 Running
2. 再看 Public IP
3. 再看 NSG
4. 再看账号和认证
5. 最后看系统层

这会对后面写排障文档很有帮助。

## 14. 本篇总结

RDP 和 SSH 是 Azure VM 最基础的远程连接方式。

学习这一步时，真正重要的不只是“试着连一下”，而是理解：

- Windows 和 Linux 的远程连接方式不同
- 能连接成功，说明计算、网络、端口和认证基本都正常
- 连接失败时不要乱猜，要按层排查
- 远程连接测试是后续部署、运维、排障的基础入口
