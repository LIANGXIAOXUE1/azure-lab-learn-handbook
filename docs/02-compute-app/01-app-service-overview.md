# App Service 概览

## 1. 是什么

App Service 是 Azure 中非常核心的应用托管服务之一，主要用于托管：

- Web 网站
- Web API
- 后端应用
- 部分定时任务 / 部署流程相关能力

你可以先把它理解成：

> 一个不用自己管底层服务器、就可以把网站或应用部署上去运行的平台服务。

和 VM 最大的不同在于：

- VM 更像你租了一台云服务器，很多事情要自己管
- App Service 更像 Azure 帮你把应用运行环境准备好了，你主要关注应用本身

所以学习 App Service 的第一步，最重要的是先建立这个认知：

> App Service 不是“另一种 VM”，而是一种更偏平台托管的应用运行方式。

---

## 2. 解决什么问题

App Service 主要解决下面这些问题：

### 2.1 网站或 API 想快速上线
如果客户想把一个网站、后台系统或 Web API 快速部署到 Azure，App Service 往往是非常优先考虑的服务。

### 2.2 不想自己维护操作系统和 Web 服务器
如果使用 VM，通常还要自己处理：

- 操作系统维护
- 补丁更新
- IIS / Nginx / Apache
- 安全配置
- 基础运行环境管理

而 App Service 会帮你托管掉很多底层细节。

### 2.3 希望更适合应用发布和运维
App Service 通常会更方便配合：

- 部署中心
- GitHub / DevOps 发布
- 应用配置
- 自定义域名
- HTTPS
- 扩缩容
- 日志与监控

### 2.4 希望减少基础设施负担
客户很多时候真正想要的不是“有一台服务器”，而是“我的应用能稳定上线”。  
如果不需要系统级控制，App Service 往往比 VM 更省事。

---

## 3. 它和 VM 的核心区别是什么

学习 App Service 时，最重要的一个问题就是：

> 它和 VM 到底有什么区别？

你可以先用最简单的方式理解：

### VM
- 你拿到的是一台服务器
- 你自己管操作系统
- 你自己装环境
- 你自己配端口和服务
- 灵活，但运维更多

### App Service
- 你拿到的是应用运行平台
- 底层服务器由 Azure 管更多
- 你更关注代码和配置
- 更适合标准网站 / API 托管
- 控制力少一些，但更省运维

所以本质上可以理解成：

- **VM** 偏 IaaS
- **App Service** 偏 PaaS

---

## 4. 学习阶段先理解什么

### 4.1 App Service 重点是“托管应用”，不是“给你一台机器”
这是入门最关键的认知。

如果你总是用“服务器思维”看 App Service，就容易困惑：

- 为什么不能像 VM 那样随便进系统
- 为什么很多底层细节看不到
- 为什么发布方式更像“部署应用”而不是“登录机器”

因为它本来就不是以“给你系统控制权”为目标的。

---

### 4.2 App Service 更适合标准化 Web 场景
例如：

- 公司官网
- 后台管理系统
- Web API
- 简单业务系统
- 演示站点

如果客户场景本身就是标准 Web 应用，App Service 往往是优先候选。

---

### 4.3 App Service 也不是万能的
如果客户明确需要：

- 系统级控制
- 自定义底层环境
- 特殊中间件安装
- 非标准运行方式
- 特定运维脚本和系统权限

那 App Service 可能就不一定最合适，VM 或容器平台可能更适合。

---

## 5. App Service 的基本组成怎么理解

学习阶段可以先接触下面几个最基础概念：

### 5.1 App Service Plan
这是承载 App Service 计算资源的计划层。

可以简单理解为：

> 应用跑在什么规格的托管环境上。

它通常关系到：

- 计算能力
- 定价层
- 扩缩容能力
- 部署区域

---

### 5.2 Web App
这是你真正创建出来承载网站 / API 的应用实例。

可以简单理解为：

> 你部署代码的地方。

---

### 5.3 Deployment
也就是把你的应用代码发布上去。

常见方式包括：

- 本地代码部署
- GitHub 部署
- ZIP Deploy
- CI/CD

---

### 5.4 Application Settings
也就是应用配置，例如：

- 数据库连接字符串
- 环境变量
- API Key
- 自定义配置项

这是后面应用上线非常常见的一部分。

---

## 6. App Service 常见适用场景

学习阶段先记住这些高频场景：

- 公司官网
- 管理后台
- REST API
- 小型到中型 Web 应用
- 演示环境
- 内部业务系统
- 对外开放的标准 Web 服务

对客户来说，App Service 常常对应的问题是：

- 网站怎么快速上 Azure
- API 怎么托管
- 我们是不是一定要买 VM
- 能不能少管服务器
- 怎么做持续部署

---

## 7. 怎么测试

学习 App Service 概览阶段，建议先做“认知测试”，不急着一开始做很复杂的部署。

---

### 7.1 测试 1：在 Portal 中创建一个最基础的 Web App
观察创建流程里会出现哪些关键项，例如：

- Subscription
- Resource Group
- App Name
- Runtime stack
- Region
- App Service Plan

---

### 7.2 测试 2：观察它和 VM 创建流程的差异
重点观察：

- 没有 OS 管理细节
- 没有 RDP / SSH 作为主要入口
- 没有你去配 VNet / Public IP 才能访问的典型学习路径
- 更偏应用运行参数而不是系统参数

---
