# Azure 报价常见漏项清单

这篇文档主要解决一个非常实际的问题：

> **为什么很多 Azure 报价看起来主资源都报了，但最后还是不完整？**

答案通常很简单：

> **因为漏报了配套资源。**

新手最常见的情况不是“不会报 VM”，而是：

- 报了 VM，漏了磁盘
- 报了数据库，漏了备份
- 报了多台应用，漏了负载均衡
- 报了公网业务，漏了流量
- 报了主系统，漏了安全项

这些漏项的结果通常是：

- 报价金额偏低
- 后续频繁返工
- 客户上线后发现成本不止这些
- 销售和客户觉得报价“不完整”或“不专业”

所以这一篇的目标是：

1. 帮你识别 Azure 报价中最常见的漏项  
2. 告诉你这些项目为什么容易被漏掉  
3. 帮你建立“报价前检查清单”的习惯  
4. 让你在实际报价时少返工、少遗漏  

---

# 一、先记住一句话：主资源只是开始，配套资源才决定报价是否完整

很多人一拿到需求，第一反应就是：

- 几台 VM？
- 什么规格？
- Linux 还是 Windows？

这当然没错，但这通常只是报价的第一层。

因为客户系统要真正运行起来，往往还需要很多“不是第一眼就能看到”的资源，比如：

- 磁盘
- 数据库
- 缓存
- 负载均衡
- 公网 IP
- 流量
- 备份
- 安全
- 存储
- CDN

所以你要逐渐建立一个习惯：

> **报完主资源后，必须再做一轮“漏项检查”。**

---

# 二、Azure 报价最常见的 10 大漏项

下面这些，是实际工作里最容易漏掉的项目。

---

# 1. 系统盘和数据盘

这是最常见的漏项之一。

---

## 为什么容易漏

因为客户在描述服务器需求时，通常只会说：

- 4核16G
- 8核32G
- 要 3 台服务器

他们很少会主动说：

- 系统盘多大
- 要不要数据盘
- 数据盘多大
- 什么性能等级

所以很多新手只报了 VM 本身，却忘了磁盘。

---

## 什么时候最容易漏

尤其下面这些场景特别容易漏：

- 数据库服务器
- 应用服务器
- 文件服务器
- 日志服务器
- 客户只说“几台机器”时

---

## 你应该怎么检查

至少问清楚：

- 每台 VM 系统盘多大
- 是否需要额外数据盘
- 数据盘多大
- 是标准 SSD 还是高级 SSD

---

## 一句话提醒

> **报 VM 时，几乎一定要顺手检查磁盘。**

---

# 2. 数据库

这也是非常高频的漏项。

---

## 为什么容易漏

因为客户常常只说：

- 我要一个系统
- 我要一个官网
- 我要一个后台
- 我要 2 台应用服务器

但没有主动提数据库。

可实际上，只要系统有：

- 登录
- 后台管理
- 用户信息
- 内容发布
- 商品订单
- 权限配置

通常就需要数据库。

---

## 什么时候最容易漏

- 带后台的官网
- 管理系统
- ERP / OA / CRM
- 电商系统
- APP 后端
- 文件管理平台

---

## 你应该怎么检查

问自己：

- 这个系统有没有业务数据？
- 有没有用户、权限、订单、表单、配置？
- 如果有，这些数据存哪里？

如果答案是“需要存”，那大概率就要数据库。

---

## 一句话提醒

> **只要不是纯静态页面，就要怀疑是否需要数据库。**

---

# 3. Redis / 缓存

这不是每个项目都必须有，但在很多业务系统里很常见。

---

## 为什么容易漏

因为很多客户不会直接说“我要 Redis”，  
但实际系统可能已经依赖它来做：

- 登录状态
- Session
- 验证码
- 热点数据缓存
- 降低数据库压力

如果是老系统迁移上云，Redis 也可能原本就存在，只是客户没主动说出来。

---

## 什么时候建议特别检查

- 电商系统
- APP 后端
- 高并发接口系统
- 登录态较多的业务
- 需要提升访问速度的系统

---

## 你应该怎么检查

可以问：

- 系统是否已有 Redis
- 是否有缓存需求
- 是否有登录状态 / 会话 / 热点数据场景

---

## 一句话提醒

> **并发稍高、读多写少、登录态多的系统，建议检查 Redis 需求。**

---

# 4. Public IP / 公网访问相关资源

很多项目是需要对外访问的，但这部分也很容易漏。

---

## 为什么容易漏

因为客户会说：

- 要上官网
- 要上商城
- 要上 APP 接口
- 要做公网访问

但不一定会细说公网入口怎么做。

如果你不主动补充，可能会漏掉：

- Public IP
- DNS
- HTTPS 证书
- 防火墙入口控制
- Application Gateway / Load Balancer

---

## 什么时候最容易漏

- 官网
- 商城
- 对外 API
- APP 后端
- 下载站点
- 图片或文件访问平台

---

## 你应该怎么检查

问自己：

- 这个系统要不要给公网用户访问？
- 如果要，入口是什么？
- 是单机公网 IP，还是走负载均衡，还是要网关？

---

## 一句话提醒

> **只要系统是对外开放的，就要检查公网访问相关资源。**

---

# 5. 负载均衡（Load Balancer / Gateway）

多台应用服务器时，这项特别容易漏。

---

## 为什么容易漏

因为客户会说：

- 我要 2 台 Web
- 我要 3 台应用服务器

但他们不一定会主动提：

- 访问怎么分发？
- 有没有高可用？
- 要不要统一入口？

所以很多人只报了多台 VM，没有报负载均衡。

---

## 什么时候最容易漏

- 2 台及以上 Web / API 服务器
- 生产环境
- 对外业务系统
- 希望高可用的系统

---

## 你应该怎么检查

问：

- 多台应用服务器是否需要流量分发？
- 是否需要统一入口？
- 是否要求一台故障后业务还能继续？

如果答案是要，大概率就要负载均衡。

---

## 一句话提醒

> **多台应用服务器，默认就要检查是否需要负载均衡。**

---

# 6. 备份（Backup）

这项在生产环境里尤其容易被忽略，但又非常重要。

---

## 为什么容易漏

因为客户最开始通常只关注：

- 系统能不能跑
- 配置够不够
- 价格高不高

但不会第一时间想到：

- 数据丢了怎么办
- 服务器坏了怎么办
- 误删了怎么恢复

所以如果你不主动问，备份很容易被省略。

---

## 什么时候最容易漏

- 生产环境
- 数据库
- 订单系统
- 管理系统
- 文件系统
- 公网业务系统

---

## 你应该怎么检查

至少确认：

- VM 要不要备份
- 数据库要不要备份
- 保留多久
- 是否要异地灾备

---

## 一句话提醒

> **生产环境几乎都应该检查备份，尤其数据库和核心业务。**

---

# 7. 安全项（Defender / Firewall / DDoS / Key Vault）

安全项经常不是第一眼看见的资源，所以特别容易漏。

---

## 为什么容易漏

因为客户很多时候不会直接说：

- 我要 Defender
- 我要 Firewall
- 我要 DDoS
- 我要 Key Vault

但如果系统是公网暴露的、或者有合规要求，这些就很重要。

---

## 什么时候特别要检查

- 公网业务
- 电商系统
- 金融 / 医疗 / 政企
- 客户对安全敏感
- 有证书、密钥、密码管理要求

---

## 你应该怎么检查

问：

- 是否需要基础安全防护
- 是否要做网络访问控制
- 是否担心公网攻击
- 证书和密钥放哪里管理

---

## 一句话提醒

> **公网业务不一定全部都买高级安全，但至少要评估安全项。**

---

# 8. 流量费用（尤其出网流量）

这是最容易在最终账单里和报价产生差异的地方之一。

---

## 为什么容易漏

因为很多人做报价时习惯只看“固定资源费”，比如：

- VM
- Disk
- Database

但像下面这些经常是按使用量收费的：

- 出网流量
- CDN 流量
- 存储访问量
- 备份容量增长

如果完全不提，客户后面很容易说：

- 为什么账单比报价高？
- 为什么还有额外费用？

---

## 什么时候最容易漏

- 官网
- 电商
- 图片站
- 下载站
- APP
- 跨境业务
- 视频、文件分发类业务

---

## 你应该怎么检查

至少要确认一个大概范围：

- 月访问量
- 大概出网流量
- 静态资源多不多
- 是否有图片 / 视频 / 下载

如果实在没有数据，也要备注：

- 流量类费用暂未包含 / 按估算值计算

---

## 一句话提醒

> **只要系统有公网访问，就要想到流量费。**

---

# 9. Blob Storage / Files / 对象存储

很多业务不只是“服务器 + 数据库”，还会有很多文件数据。

---

## 为什么容易漏

因为客户通常只会说：

- 我们有图片
- 有附件
- 有导出文件
- 有日志
- 有共享文件

但不会主动说要用 Blob 还是 Files。

这时就需要你自己判断是否要补存储服务。

---

## 什么时候特别容易漏

- 官网图片
- 电商商品图
- 用户上传附件
- 文件平台
- 日志归档
- 报表导出
- 视频或音频文件

---

## 你应该怎么检查

问：

- 系统里是否有图片、附件、导出文件？
- 这些文件打算放哪？
- 是否需要共享文件夹？
- 是否需要长期归档？

---

## 一句话提醒

> **只要系统涉及文件，就要检查是不是需要独立存储服务。**

---

# 10. 监控、日志、运维类资源

这类资源不是每个报价都会单独展开，但在很多生产项目里不能完全忽略。

---

## 为什么容易漏

因为客户通常先关心“系统能不能部署”，  
不一定会第一时间提：

- 监控
- 日志分析
- 告警
- 运维审计

但系统真的上线后，这些会很快变得重要。

---

## 什么时候建议检查

- 生产环境
- 多台服务器
- 重要业务系统
- 需要审计或排障能力的项目

---

## 可能涉及的资源

- Azure Monitor
- Log Analytics
- Alerts
- Defender 相关日志
- 审计日志存储

---

## 一句话提醒

> **小项目可以先简化，但正式生产系统最好考虑基本监控与日志。**

---

# 三、最容易漏项的 5 类场景

下面这些场景是“高风险漏项区”，你做报价时建议重点检查。

---

# 场景 1：客户只说“我要几台服务器”

这是最容易出错的情况。

例如客户说：

- 我要 3 台服务器
- 2 台 4核16G
- 1 台 8核32G

你不能只报 3 台 VM。

还要继续检查：

- 磁盘
- 数据库
- 公网访问
- 负载均衡
- 备份
- 安全
- 流量

---

# 场景 2：公网 Web / APP / 电商系统

这类系统经常会漏：

- Public IP
- Load Balancer
- CDN
- 流量
- 安全项
- 备份

因为它们不只是“跑起来”，还要考虑：

- 用户访问
- 并发分发
- 静态资源加速
- 攻击防护
- 恢复能力

---

# 场景 3：数据库单独上云

这类项目经常漏：

- 数据盘
- 备份
- 高可用
- 监控
- 网络访问控制

因为大家容易只看“数据库实例本身”，忽视配套。

---

# 场景 4：文件 / 图片 / 附件类系统

这类项目经常漏：

- Blob Storage / Files
- 出网流量
- CDN
- 生命周期管理
- 备份

---

# 场景 5：生产环境

只要是生产环境，就要重点检查：

- 备份
- 安全
- 监控
- 负载均衡
- 高可用
- 性能余量

测试环境可以简化，生产环境不能只按“最低可运行标准”报价。

---

# 四、为什么新手最容易漏这些项

你会发现漏项并不是因为你笨，而是因为这些资源有几个共同特点：

---

## 1）客户不会主动说
客户只会说“我要系统”，不一定会说“我要备份”。

---

## 2）这些资源通常不是系统主角
客户更容易先想到服务器、数据库，不会先想到安全和监控。

---

## 3）很多费用是按量而不是固定
例如流量、存储、备份容量，所以更容易被忽略。

---

## 4）配套项跨多个分类
例如一个公网系统同时涉及：
- 网络
- 安全
- 流量
- CDN
- 证书

新手如果没有体系，就很容易漏其中几项。

---

# 五、最实用方法：报价完成后，按 5 类做一次反查

每次报价做完后，你都可以用下面这个办法快速复核。

---

## 1）计算
- VM / App Service / AKS 报了吗？
- 数量对吗？
- 规格对吗？

## 2）存储
- 系统盘报了吗？
- 数据盘报了吗？
- 文件 / 图片 / 日志存储报了吗？

## 3）数据库
- 数据库报了吗？
- Redis 有无需要？
- 数据层是否完整？

## 4）网络
- 公网入口报了吗？
- Public IP / Load Balancer / CDN / VPN 是否需要？
- 流量是否考虑？

## 5）安全与备份
- Backup 报了吗？
- Defender / Firewall / DDoS 是否需要？
- 有无证书 / 密钥管理需求？

这个方法很简单，但特别有效。

---

# 六、报价前 1 分钟自检清单

下面这份是最适合新手直接照着过的版本。

---

## Azure 报价前 1 分钟自检清单

### 计算
- [ ] 服务器 / 应用运行资源已确认
- [ ] 规格、数量、OS 已确认

### 磁盘与存储
- [ ] 每台系统盘已确认
- [ ] 每台数据盘已确认
- [ ] 是否有 Blob / Files / 附件 / 图片存储需求已确认

### 数据层
- [ ] 是否需要数据库已确认
- [ ] 数据库类型已确认
- [ ] 是否需要 Redis 已确认

### 网络层
- [ ] 是否需要公网访问已确认
- [ ] 是否需要 Public IP 已确认
- [ ] 是否需要负载均衡已确认
- [ ] 是否需要 CDN 已确认
- [ ] 是否有流量估算或流量备注已确认

### 安全与备份
- [ ] 是否需要备份已确认
- [ ] 是否需要 Defender / Firewall / DDoS 已确认
- [ ] 是否有密钥 / 证书管理需求已确认

### 输出方式
- [ ] 按量 / RI 方式已确认
- [ ] 月度 / 年度输出方式已确认
- [ ] 备注和前提假设已写明

# 七、如果实在来不及，至少别漏这 8 项

如果你时间特别紧，至少优先检查下面 8 项：

1. VM 本身  
2. 系统盘  
3. 数据盘  
4. 数据库  
5. Public IP / 公网访问  
6. 负载均衡（多台应用时）  
7. 备份  
8. 流量 / 备注说明  

这 8 项能帮你避开最常见的大坑。

# 八、信息不全时，怎么避免“假完整报价”

有时候销售会催得很急，客户信息又不全。  
这时最危险的不是“先出一版”，而是：

> **出了一版看起来完整、实际前提没说明的报价。**

更合理的做法是下面这样。

---

## 做法 1：先报已确认部分
例如：
- VM
- Disk
- 基础数据库
- 基础网络

---

## 做法 2：明确列出未包含项
例如备注：

### 当前未包含 / 待确认项
- 出网流量
- CDN
- 备份保留策略
- 高级安全服务
- 日志与监控

---

## 做法 3：写清假设条件
例如：

### 报价前提假设
- 假设区域为 East US
- 假设应用服务器使用 Linux
- 假设每台包含标准 SSD 数据盘
- 流量费用暂未纳入，需根据访问量另行估算

这样即使信息不全，你的报价也不会显得不专业。

# 九、这篇你要记住的核心结论

## 1. Azure 报价最常见的问题不是主资源不会报，而是配套项漏报
VM 报
