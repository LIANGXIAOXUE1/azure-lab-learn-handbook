
# Subscription / Resource Group / Region

## 1. 是什么

在 Azure 里，资源不是“随便创建”的。  
当你创建虚拟机、存储账号、数据库、虚拟网络等资源时，通常都要先明确三件事：

- 这个资源属于哪个 **Subscription（订阅）**
- 这个资源放在哪个 **Resource Group（资源组）**
- 这个资源部署到哪个 **Region（区域）**

这三个概念是 Azure 最基础的资源管理和部署逻辑。

---

### 1.1 Subscription（订阅）

Subscription 是 Azure 中的重要管理边界，可以理解为：

> Azure 资源归属、计费、配额和权限控制的重要范围。

当你在 Azure 中创建资源时，这些资源通常都属于某一个订阅。

订阅通常和下面这些事情有关：

- 费用记到哪里
- 资源配额如何计算
- 哪些人有权限操作这些资源
- 某些服务是否可用
- 管理范围如何划分

你可以先把它理解成：

> “这一批资源归谁管、算谁的账、谁能操作”的基础范围。

---

### 1.2 Resource Group（资源组）

Resource Group 是 Azure 中用于组织资源的逻辑容器。

可以理解为：

> 把一组相关资源放在一起管理的地方。

例如，一个项目可能会包含：

- 虚拟机
- 虚拟网络
- 公网 IP
- 存储账号
- 数据库

你可以把这些资源放进同一个 Resource Group 中，方便统一查看、管理、打标签、授权和删除。

资源组更多解决的是“怎么组织资源”的问题，而不是“怎么计费”的问题。

---

### 1.3 Region（区域）

Region 是 Azure 数据中心所在的地理区域。

你在 Azure 创建资源时，很多服务都需要选择一个 Region，例如：

- East US
- West Europe
- Southeast Asia
- Japan East

可以简单理解为：

> Region 就是资源实际部署到哪个地理位置的数据中心区域。

它会影响很多实际问题，比如：

- 访问延迟
- 服务是否可用
- 成本
- 合规要求
- 灾备设计

所以 Region 不是一个随便选的字段，而是非常基础的部署决策之一。

---

## 2. 解决什么问题

### 2.1 Subscription 解决什么问题

Subscription 主要解决下面几类问题：

#### 计费归属
Azure 中大多数资源的费用最终都会归属到某个订阅下。

#### 配额边界
一些 Azure 服务的使用配额是以订阅为边界管理的。

#### 权限控制边界
权限经常可以在订阅层面分配，比如谁可以查看、创建、删除资源。

#### 管理范围划分
企业里可能会有多个团队、多个部门、多个环境，订阅可以作为更高一级的管理边界。

---

### 2.2 Resource Group 解决什么问题

Resource Group 主要解决的是资源组织和生命周期管理问题。

#### 资源分类管理
把同一个项目、同一个实验、同一个环境的资源放在一起，便于查看和维护。

#### 生命周期统一管理
如果一个实验结束了，删除整个资源组，里面的资源通常可以一起删除。

#### 项目维度组织资源
例如：

- 测试环境一个资源组
- 生产环境一个资源组
- 某个客户项目一个资源组
- 某个练习实验一个资源组

所以资源组特别适合做“逻辑分组”和“统一管理”。

---

### 2.3 Region 解决什么问题

Region 主要解决的是：

#### 资源部署到哪里
Azure 资源不是“漂在空中”的，它们最终都要部署在某个实际区域中。  
Region 决定了这些资源主要落在哪个地理位置。

#### 降低访问延迟
如果用户主要在亚洲，就把资源部署到更接近亚洲用户的区域，通常访问体验会更好。  
如果部署得很远，网络延迟可能会更高。

#### 满足合规和数据驻留要求
有些客户会关心：

- 数据放在哪个国家或地区
- 是否必须放在本地或指定区域
- 是否满足监管要求

这时候 Region 的选择就很重要。

#### 影响服务可用性
并不是所有 Azure 服务都在所有 Region 中可用。  
某些服务可能：

- 某个区域不支持
- 某个区域暂时没有容量
- 某个区域支持的功能不完整

#### 影响架构设计
如果你要考虑高可用、备份、容灾、跨区域部署，Region 会直接影响设计方式。

---

## 3. 三者关系怎么理解

可以先用一句话记住：

- **Subscription**：资源归哪个管理和计费边界
- **Resource Group**：资源放在哪个逻辑容器里管理
- **Region**：资源实际部署到哪个地理区域

创建一个 Azure 资源时，通常至少要先考虑这三件事：

1. 这个资源放在哪个 Subscription
2. 这个资源放在哪个 Resource Group
3. 这个资源部署到哪个 Region

可以简单理解为：

- Subscription 解决“归谁管、算谁的账”
- Resource Group 解决“和哪些资源放在一起”
- Region 解决“资源实际部署在哪”

一个简单示意如下：

```text
Subscription
└─ Resource Group
   ├─ VM（Japan East）
   ├─ Storage Account（Japan East）
   └─ SQL Database（Southeast Asia）
```

---

## 4. 一个容易理解的类比

如果把 Azure 想成一个大公司里的资产管理体系，可以这样类比：

### Subscription 像什么？
像一个更高层级的“账户 / 账本 / 管理范围”。

它决定：

- 账算到哪里
- 谁有权限
- 配额如何分配
- 这一批资源归哪个管理边界

### Resource Group 像什么？
像这个账本下面的“项目文件夹”。

它决定：

- 哪些资源放在一起
- 哪一批资源属于同一个项目或环境
- 是否可以统一删除和统一管理

### Region 像什么？
像这些资源实际放在哪个城市或机房。

它决定：

- 资源离用户远不远
- 访问速度怎么样
- 某些服务能不能用
- 是否满足数据驻留和合规要求

所以可以简单记成：

- **Subscription**：偏“账”和“管理边界”
- **Resource Group**：偏“组织”和“生命周期”
- **Region**：偏“部署位置”和“访问体验”

---

## 5. 基础关系怎么理解

你可以先建立这样一个简单认知：

- 一个订阅下面可以有多个资源组
- 一个资源组里可以有多个资源
- 一个资源只能属于一个资源组
- 同一个资源组里的资源不一定在同一个区域
- 资源在创建时通常需要先选好订阅、资源组和区域

一个简单结构示意：

```text
Subscription
├─ Resource Group A
│  ├─ VM（Japan East）
│  ├─ VNet（Japan East）
│  └─ Storage Account（Japan East）
│
└─ Resource Group B
   ├─ SQL Database（Southeast Asia）
   ├─ App Service（Southeast Asia）
   └─ Key Vault（Southeast Asia）
```

## 6. 怎么创建和选择

### 6.1 创建资源组步骤

1. 登录 Azure Portal
2. 搜索 `Resource groups`
3. 点击 `Create`
4. 选择一个 Subscription
5. 输入资源组名称
6. 选择资源组元数据所在区域
7. 点击 `Review + create`
8. 点击 `Create`

---

### 6.2 创建时要注意什么

#### 资源组名称要有规则
建议命名尽量清晰，能看出用途，例如：

```text
rg-azure-learning-test
rg-project-a-dev
rg-project-a-prod
```

#### 先想清楚资源放哪一组
不要所有资源都随便堆在一个资源组里。  
最好按项目、环境、实验、团队来规划。

#### 资源组区域不要和资源区域混淆
资源组本身要选一个区域，但这不代表组内所有资源必须都部署在同一个区域。

---

### 6.3 Region 怎么选

初学阶段可以先按下面几个原则选：

#### 优先选离用户或团队更近的区域
如果你主要是自己做学习实验，通常可以优先选择：

- 离自己更近
- Portal 中可用性较好
- 资源创建更稳定

的区域。

#### 优先选服务支持更完整的区域
某些常见区域的服务支持会更完整。  
如果你发现某个服务在当前区域不可用，可以尝试换到另一个更常见的区域。

#### 学习实验尽量集中
初学阶段建议不要一会儿 East US、一会儿 West Europe、一会儿 Japan East，容易把资源分散得太乱。  
可以先选 1～2 个常用区域，保持练习一致性。

#### 生产环境要结合业务和合规要求
如果是正式项目，选 Region 不能只看“顺手”，还要结合：

- 用户分布
- 法规要求
- 成本
- 灾备需求
- 服务可用性

---

## 7. 怎么测试

学习 Subscription、Resource Group、Region 时，建议做一个最小实验。

### 测试步骤

1. 创建一个新的 Resource Group
2. 在该资源组中创建一个简单资源，例如：
   - Storage Account
   - Virtual Machine
3. 创建资源时观察：
   - 资源属于哪个 Subscription
   - 放入了哪个 Resource Group
   - 选择了哪个 Region
4. 打开资源组页面，确认该资源已经归属到这个组中
5. 查看资源组中的资源列表
6. 测试完成后删除整个资源组
7. 验证组内资源是否被一起删除

---

### 这个测试想让你理解什么

这个测试重点不是学会点按钮，而是理解三件事：

#### 第一，资源不是孤立存在的
它们通常属于某个订阅，也属于某个资源组。

#### 第二，资源组是生命周期管理单位
很多时候删除一个资源组，就等于把这个实验环境、测试环境、项目临时资源整体删除。

#### 第三，资源一定部署在某个区域
创建资源时的 Region 不是装饰项，而是部署决策的一部分。

---

## 8. 学习这部分时最重要的认知

### 8.1 创建资源前，先想清楚放在哪个订阅
因为订阅通常关系到：

- 成本归属
- 权限边界
- 配额
- 管理方式

### 8.2 创建资源前，先想清楚放在哪个资源组
因为资源组关系到：

- 资源怎么分类
- 后续怎么找
- 是否方便统一删除
- 是否方便按项目或环境管理

### 8.3 创建资源前，先想清楚部署到哪个区域
因为区域关系到：

- 延迟
- 服务可用性
- 成本
- 合规
- 灾备设计

### 8.4 不要把“订阅”“资源组”“区域”看成同一个东西
这三个概念经常一起出现，但解决的问题不同：

- 订阅偏“上层管理边界”
- 资源组偏“资源组织与生命周期管理”
- 区域偏“资源部署位置与可用性”

---

## 9. 常见问题

### 问题 1：资源组和订阅有什么区别？

订阅更偏向计费、权限、配额边界；  
资源组更偏向资源组织和生命周期管理。

你可以简单理解为：

- **订阅**：更上层的范围，管账、管权限、管配额
- **资源组**：更具体的分组，管项目、管资源归类、管统一删除

---

### 问题 2：同一个资源组里的资源必须在同一个区域吗？

不一定。  
资源组本身有自己的元数据区域，但资源组里的资源可以部署在不同区域。

不过在实际设计中，很多资源还是会尽量放在相近区域，以减少网络延迟、避免架构混乱，也方便管理。

---

### 问题 3：删除资源组会怎样？

删除资源组时，通常会删除该资源组中的所有资源。  
所以这个操作一定要非常小心。

这也是为什么很多测试实验会单独放到一个资源组里：  
这样实验结束后可以整体清理，不容易漏删。

---

### 问题 4：一个资源能同时属于多个资源组吗？

不能。  
一个 Azure 资源只能属于一个资源组。

所以在创建资源之前，最好先想清楚这个资源应该放在哪个组里。

---

### 问题 5：一个项目是不是一定对应一个订阅？

不一定。  
这要看企业的管理方式、权限划分、预算控制和组织规模。

有些公司会：

- 一个项目一个订阅
- 一个部门一个订阅
- 一个环境一个订阅
- 或多个项目共用一个订阅

没有绝对唯一标准，但核心原则是：  
要有利于计费、权限、配额和管理边界控制。

---

### 问题 6：测试环境和生产环境要不要分开？

通常建议分开，至少在资源组层面分开，很多情况下也会在订阅层面分开。

因为这样更方便：

- 权限隔离
- 成本区分
- 风险控制
- 生命周期管理
- 避免误删生产资源

---

### 问题 7：Region 是不是资源所在的位置？

是的。  
大多数情况下，Region 可以理解为这个资源主要部署在哪个 Azure 地理区域。

---

### 问题 8：资源组的区域和资源的区域一样吗？

不一定。  
资源组有自己的元数据区域，但资源本身可以部署到其他区域。

所以创建资源时，不要以为资源组选了某个区域，资源就只能跟着那个区域。

---

### 问题 9：为什么有些服务在某个区域不能创建？

常见原因包括：

- 该区域不支持该服务
- 该区域不支持该规格
- 该区域当前容量不足
- 该区域暂未开放某些功能
- 当前订阅在该服务上有额外限制

---

### 问题 10：Region 选错了怎么办？

很多资源创建后不能直接随便改 Region。  
有些情况下需要：

- 重新部署
- 迁移数据
- 新建资源再切换

所以正式环境里，Region 最好一开始就想清楚。

---

## 10. 客户常问的问题

这一部分重点不是只记录客户怎么问，而是练习你该怎么回答。

回答顺序建议按这个逻辑：

1. 先讲价值
2. 再讲能力
3. 最后根据客户管理场景展开

---

### 问题 1：订阅是什么？是不是一个项目一个订阅？

**简短回答：**  
订阅可以理解为 Azure 里比较上层的管理和计费边界，但不一定必须一个项目一个订阅。

**展开回答：**  
订阅主要影响的是：

- 费用归属
- 权限边界
- 配额管理
- 管理范围划分

有的公司会一个项目一个订阅，有的公司会一个部门一个订阅，也有的公司会测试和生产分不同订阅。  
到底怎么分，不是固定规则，而是看企业怎么做管理、预算和权限隔离。

如果客户规模比较小，前期可能不需要分得太细；  
如果客户项目多、团队多、权限复杂，就更可能需要多个订阅来做隔离。

---

### 问题 2：资源组怎么划分比较合理？

**简短回答：**  
资源组一般按项目、环境、系统或实验来划分，核心原则是方便管理和控制生命周期。

**展开回答：**  
最常见的划分方式包括：

- 按项目划分
- 按环境划分（dev / test / prod）
- 按实验划分
- 按系统模块划分

例如：

- `rg-project-a-dev`
- `rg-project-a-prod`
- `rg-learning-vm-lab`

如果客户的目标是方便统一删除、统一查看、统一授权，那么资源组划分就应该围绕这些目标来设计。

---

### 问题 3：测试环境和生产环境要不要分开？

**简短回答：**  
通常建议分开，至少资源组分开，很多场景下订阅也建议分开。

**展开回答：**  
这样做的好处包括：

- 减少误操作风险
- 权限更清晰
- 成本更容易区分
- 资源生命周期更好控制
- 出问题时不容易互相影响

如果客户比较重视规范管理、安全隔离和成本管理，分开通常是更合理的做法。

---

### 问题 4：删除资源组是不是就全部删掉了？

**简短回答：**  
通常是的。删除资源组时，组内资源一般会一起删除。

**展开回答：**  
这也是资源组非常适合作为实验环境和临时环境管理单位的原因。  
但对于正式环境来说，这也意味着必须非常谨慎，因为误删资源组可能导致整套资源一起被删除。

所以对客户要强调：

> 资源组方便统一管理，也方便统一删除，但正因为如此，生产环境操作一定要更谨慎。

---

### 问题 5：权限是给资源组还是给整个订阅？

**简短回答：**  
两种都可以，要看你希望权限控制到什么范围。

**展开回答：**  
如果希望一批资源统一授权，可以在资源组层面授权；  
如果希望更大范围统一控制，也可以在订阅层面授权。

一般来说：

- **订阅层面授权**：范围更大
- **资源组层面授权**：更细、更灵活

对于客户来说，可以这样理解：

> 如果一个团队只负责某个项目，就可以给它某个资源组的权限；  
> 如果一个平台管理员需要看管整个订阅下的资源，就可以在订阅层面授权。

---

### 问题 6：为什么创建资源时一定要选区域？

**简短回答：**  
因为 Azure 资源需要实际部署在某个数据中心区域，区域会影响性能、可用性、成本和合规。

**展开回答：**  
Region 不是一个随便填的选项，它决定了资源部署在哪里。  
这会影响用户访问延迟、某些服务是否可用、成本差异，以及数据是否满足合规要求。

如果客户后续还要考虑高可用或灾备，区域选择会更重要。

---

### 问题 7：是不是所有资源都应该放在同一个区域？

**简短回答：**  
不一定，但很多相关资源通常会尽量放在相近区域，以降低延迟和简化管理。

**展开回答：**  
从技术上说，资源可以分布在不同区域。  
但如果一个网站、数据库、存储彼此频繁交互，通常会尽量放在接近的区域，否则可能带来：

- 网络延迟
- 架构复杂度提高
- 成本增加
- 排障难度上升

所以是否放同一区域，要看业务关系和架构目标。

---

### 问题 8：为什么某个服务在这个区域没有？

**简短回答：**  
因为 Azure 不同服务在不同区域的支持情况不完全一样。

**展开回答：**  
常见原因包括：

- 这个区域还没有开放该服务
- 这个区域不支持该规格
- 该区域容量紧张
- 某些新功能先在部分区域开放

所以在做方案时，通常需要同时看：

- 区域是否支持服务
- 区域是否支持对应规格
- 是否满足客户性能和合规要求

---

## 11. 你可以怎么对客户解释

如果客户第一次接触这三个概念，不建议一开始就讲太抽象的术语。  
更适合先从“管理、分类、部署位置”三个角度解释。

可以这样说：

> 在 Azure 里，订阅更像是上层的管理和计费范围，决定资源算到哪里、谁能管、权限怎么分。  
> 资源组更像是具体项目或环境的资源文件夹，用来把同一套资源放在一起管理。  
> 区域则是这些资源实际部署到哪个数据中心位置，会影响访问速度、服务可用性和合规要求。  
> 所以一般创建资源时，要先确定归哪个订阅、放到哪个资源组、部署到哪个区域。  
> 这样后面做权限控制、成本查看、环境区分、统一删除和架构设计时都会更清楚。

## 14. 本篇总结

Subscription、Resource Group、Region 是 Azure 中最基础、也最常用的三个概念。

可以先记住一句话：

- **Subscription**：偏计费、权限、配额、管理边界
- **Resource Group**：偏资源组织、分类、生命周期管理
- **Region**：偏资源部署位置、延迟、可用性和合规

学习 Azure 时，后面创建几乎所有资源都会反复遇到它们。  
所以这一步不是小知识，而是后续所有实验和项目实践的基础。
